1. 簡介：Skills 是 LLM Agents 的程序記憶

Agent Skills 是一種把知識、公司脈絡與可重複流程交給代理人的方法。最小單位是一個資料夾，裡面必須有 SKILL.md，並可選擇加入 scripts/、references/、assets/ 等目錄。

它解決四個核心問題：指令太多導致 context rot、模型缺少「怎麼做」的 procedural memory、多代理人架構過度膨脹，以及跨平台可攜性不足。

核心命題

Skills 不是把所有專家知識塞進 prompt，而是把可重複流程封裝成按需載入的工程單位。這讓一個通用代理人可以在需要時變成專家，且不讓每一輪對話都背負完整上下文成本。

2. 什麼是 Agent Skill

Agent Skill 是一種讓通用代理人按需取得專家能力的 primitive。它可以只是單一 Markdown 檔，但真正價值在於 progressive disclosure：metadata 永遠可見，body 觸發後才載入，references、assets、scripts 只有需要時才使用。

cafe-preparation/
├── SKILL.md
├── scripts/
│   ├── calc_quantities.py
│   └── convert_to_ingredients.py
├── references/
│   ├── menu_and_recipes.md
│   └── minimums.md
└── assets/
    ├── prep_sheet_template.md
    └── shopping_list_template.md

兩條建立路徑

Path A 是把既有知識翻譯成 Skill。domain experts 已有 runbooks、SOP、onboarding guides，只需要整理成代理人能觸發與使用的格式。

Path B 是把代理人剛成功完成的任務結晶化。代理人根據成功 trajectory 提出 SKILL.md 草稿，人類 review 後納入 library，這就是 meta-skills 的入口。

---
name: cafe-preparation
description: |
  Calculates daily ingredient needs and generates prep sheets for cafe operations.
  Use when the user asks to estimate daily quantities, convert drinks to ingredients,
  or generate shopping lists. Do NOT use for employee shift scheduling or financial accounting.
---

一開始最值得投資的是 naming 與 description。description 是 routing algorithm，模型只看這段來判斷是否載入 skill。

Skill、MCP、AGENTS.md

Skill vs MCP：MCP 負責 reach，把 agent 連到外部系統；Skill 負責 know-how，教 agent 如何完成某類工作。兩者不是競爭，而是組合。

Skill vs AGENTS.md：AGENTS.md 是專案常駐上下文；Skills 按需載入。乾淨做法是兩者並用，讓 AGENTS.md 保持精簡，必要時列出 skill catalog。

3. 為什麼 Agent Skills 這麼快流行

2025 年初，許多團隊會用 multi-agent architecture 解決 back-office 自動化：一個 router agent 加上多個 specialist sub-agents。這能工作，但帶來 CI/CD、orchestration、deployment surface 與 regression 的維護成本。

Agent Skills 把這件事變輕：一個 agent 搭配 skill library，skills 可跑 commands、呼叫 MCP servers、包 Python scripts，由 agent 決定何時載入。你維護的是 skills，不是大量 agents。

方案問題何時適合
一個巨大 context window立刻造成 context rot 與 token cost幾乎不建議
RAG over runbooks要維護 vector DB、embedding、chunking以查詢知識為主，而非程序流程
一流程一 sub-agentdeployment 與 evaluation surface 暴增需要平行性、權限隔離或異質模型
一 agent + 多 skills需嚴格評估 routing 與 overlap大量流程變體、同一 agent 可按需專精

4. 評估 Skills：沒有測試的 Skill 只是希望

SkillsBench 研究顯示，19% 的真實任務在加入 skill 後反而更差。這些失敗可預測，主要分成四類：Trigger Failure、Execution Failure、Token Budget Failure、Regression。

Evaluation Toolkit

五種互補測試模式覆蓋 Agent Skills 的主要 failure surface。

PatternDescriptionFailure ModeWhen Required
Eval-as-Unit-Testskill 每次變更都在 CI 中跑測試；失敗阻擋 merge。AllEvery skill, every change
Golden Dataset與 skill 一起版本化的 input / expected output pairs。Execution, TriggerDraft tier and above
LLM-as-Judgepeer model 依 rubric 評估輸出，交換 reference / actual 位置消除 bias。ExecutionRead-only and draft
Adversarial / Red-Team用 rephrasing 與 negative boundary cases 系統性探測失敗。Trigger, ExecutionBefore action-allowed graduation
Canary / Shadow Modefull rollout 前用 shadow 或 1% live traffic 受控觀察。RegressionBefore each action-allowed release

Single-Skill Sub-Agent pattern

Trajectory testing 評估的是 host agent + skill 的複合系統。若多 skill trajectory 失敗，很難拆分是 routing、instruction quality 或 execution fidelity。因此校準時建議先用 Agent + 1 Skill 對比 Base Agent。

trigger 是第一道 gate

一個永遠不觸發的 skill 幫不上忙；一個觸發太廣的 skill 則會注入無關 context。Vercel 的 production analysis 發現，預期穩定啟用的 skills 有 56% non-invocation rate；設計不良的 skill 甚至可能比沒有 skill 更差。

  • Testable specificity：寫出 3 個 positive triggers 與 3 個 negative triggers。
  • Clarity：模糊 query 不應與相鄰 skills 重疊。
  • Execution fidelity：description 描述實際能力，不是 aspirational behavior。
  • Rephrasing stability：使用者換句話說時仍能穩定 route。

Evaluation-Driven Development

EDD 要求在寫 SKILL.md 前先寫 evaluation cases，明確指定 input、expected skill、expected tool calls、expected output 與 rubric。

{
  "case_id": "refund_dup_charge_001",
  "input": "I was charged twice for order #4521 last Tuesday",
  "expected_skill": "refund_processor",
  "expected_tool_calls": [
    {"tool": "lookup_order", "args": {"order_id": "4521"}},
    {"tool": "check_duplicate_charge", "args": {"order_id": "4521"}}
  ],
  "expected_output_format": "confirmation_with_refund_id",
  "rubric": ["acknowledges duplicate", "cites order id", "provides next step"]
}

Token budget 與 co-loading

不要只在 isolation 下評估 skill。production agents 常 co-load 5 到 15 個 skills，單一 skill body 超過 5,000 tokens 時，單獨測可能正常，合載後會造成 context rot。

Trigger

正向與負向 test cases；目標 90% trigger accuracy。

Execution

代表性輸入範圍內輸出正確，tool trajectory 正確。

Regression

加入新 skill 後既有 library suite 零退步。

Token budget

與常用 skills co-load 後不退化無關 turns。

5. 從 Prototype 到 Production

上線後，模型不再是最有趣的部分；skills 是讓你可靠出貨的工程 primitive。Google Agents CLI 將 scaffold、build、evaluate、deploy、publish、observe 等 lifecycle 包成 skills，支援 ADK agent 的完整開發流程。

對 Claude Code v2.1.88 的 reverse-engineering 顯示，98.4% codebase 是 operational infrastructure，真正 agent loop 只有 1.6%。隨 foundation models 的 baseline reasoning 收斂，可靠性的差異化會來自 deterministic engineering，而在這層工程裡，被組合與重用的單位就是 skill。

Agent improvement methodologies 對照
改進方式週期失敗模式誰能做Context Tax
Model swap數天到數週無關任務 regressionML/platform team無
System prompt edit數分鐘到數小時context rot、instruction conflictprompt owner靜態，每輪都付
Fine-tune數週到數月catastrophic forgetting、overfittingML team無
New skill數小時到數天只限匹配 turns任何 domain team動態，觸發才載入

token budget 的實務含義很直接：capacity 不是正確指標，1M-token window 也可能在 50K tokens 出現顯著退化；active context 是預算，不是容器；skills 透過按需載入讓 active context 保持小，同時讓可用能力近乎無界。

6. Meta-Skills 與 Self-Improving Skills

Meta-skill 的核心是把成功 production histories 轉成可重用 procedural memories。這不是讓 LLM 無限制自我修改，而是建立「成功軌跡 → Skill 草稿 → eval gate → human review / CI」的治理迴圈。

若沒有代表性測試、production-like co-loading、ownership 與 review，meta-skill 會放大錯誤。它應是軟體工程迴圈的一部分，而不是跳過工程紀律的捷徑。

7. 組合與封裝 Skills

真實 workflow 不會永遠塞在單一 skill 裡。composition problem 包含 skills 如何互相引用、如何傳遞 state、如何避免 circular dependencies。把 raw LLM outputs 在 isolated skills 間傳來傳去通常不可靠，產業正在從 naive prompt chaining 走向可預期 orchestration。

DAG Orchestration

  • Decoupled State：DAG state routing 不依賴在 LLM prompt 中累積執行歷史。
  • File Message Bus：controller 透過 schema references 在 subagent nodes 間交接。
  • Protected Attention：把 payload 從文字 prompt 抽離，避免 context window bloat。

Capability Profiles

同時啟用所有 skills 會降低 natural language routing 並壓垮 context window。Capability Profile 是針對特定 execution state 的 tool bundle，可定義 active skills、tool access、guardrails、workflow topology 與 LLM parameters。

架構機制主要好處適合場景
Linear Pipelines固定 nodes 間依序傳文字工程負擔低單一領域、低複雜度 generative tasks
DAG Orchestrationgraph-based execution，schema refs 傳 state避免 cycles、context isolation高可靠 multi-agent workflows
Capability Profiles可替換、版本化的 parameters 與 tool bundles快速 persona switchingrole-based deployment
Shift Intelligence Left

不要在 runtime 用大量「ALWAYS DO X」逼 LLM 做 deterministic behavior。把可決定邏輯推進 scripts，把主觀判斷蒸餾成可測試 skills，減少混沌面積。

8. 如何在大量 Skills 中選擇

到 2026 年初，公開 skill marketplaces 已超過 40,000 listings。選擇問題本身已成為治理問題。

  1. vendor-specific tools 優先 first-party skills。由底層產品團隊維護的 skills 通常更正確、更穩定。
  2. pin everything。community skills 會演進，昨天可用的 unpinned dependency 明天可能壞掉。
  3. adopt 前先 audit。skill 是會在你 context 中執行的 code，必須用 supply-chain hygiene 對待。
來源信任姿態範例維護者
First-party vendor skills預設可信，但 pin versiongoogle/agents-cli、google/skills、anthropics/skills、stripe/ai底層產品團隊
Organization-curated skills組織內可信，adoption 時 reviewyour-org/retail-skills、private finance skills內部 domain teams
Community skillsadoption 前 audit，積極 pinawesome-agent-skills、SkillsMP、addyosmani/agent-skills志願者，維護程度不一

9. 結論

Agent Skill 看起來只是含 Markdown 與一些可選 scripts 的資料夾，卻正在重塑 AI Agents 的建構方式。它提供了可測試的 procedural memory，讓 foundation models 記住如何一步步執行任務。

透過 progressive disclosure，skills 解決 context rot；單一通用 agent 可以使用大量 specialist workflows，而不被 token budget 嗆住。真正有趣的工作發生在格式周圍：production-like co-loaded evaluation、避免把 context window 當 message bus 的 skill composition、從成功 traces 草擬 skill、用人類 review 取代手寫一切，以及把 institutional knowledge 編碼成版本化、可測試、可治理 library。

如果今天開始，建議很簡單：start small，從已經有的知識開始，把 Skills 當 code，衡量你交付的東西；當 Skill 足夠時，不要急著使用 multi-agent architecture。

Appendix A：實作速查表

Minimal SKILL.md

---
name: skill-name
description: |
  [What it does in one verb-led sentence.]
  Use this skill when the user [trigger phrase 1], [trigger phrase 2],
  or [trigger phrase 3]. Do NOT use for [anti-trigger 1] or [anti-trigger 2].
version: 1.0.0
license: MIT
allowed-tools: [Optional] Read Bash Write
metadata:
  author: [Optional] your-handle
---
# Skill Name

## When to use
- [Concrete scenario]

## When NOT to use
- [Out-of-scope scenario]

## Workflow
1. [Step]
2. [Step]
3. See `references/advanced.md` for [edge case].

五條規則

One skill, one job；description 是 interface；skills 是 dependencies；正確團隊 owns 正確 skill；runtime 應可替換。

品質原則

先自己跑任務、給理由、每一行都要有價值、指令可驗證、重複邏輯放 scripts。

Do

小而具體、重視 description、信任 progressive disclosure、把 deterministic work 放進 scripts、把 Skills 當 code。

Don't

模糊描述、超長 SKILL.md、hard-code secrets、把全域規則塞進 skill、未掃描即安裝第三方 skills。

Deployment checklist

  • Frontmatter validates（lint passes）。
  • Description 包含 what + when + when-not。
  • Scripts 有 unit tests 並在 CI 通過。
  • Eval suite 達 min-pass threshold。
  • Security scan clean，無 secrets、無不可信 deps。
  • Description 由作者以外的人 review。
System prompt = instinct。AGENTS.md = project README。Tools / MCP = hands。RAG = library。Skills = 經驗同事第一天交給你的 runbook，而且 AI 永遠不會忘記。

Appendix B：零售業 Vertical Skills 案例

兩家零售商即使用同一個 runtime、同類 API，購物體驗仍可能完全不同，因為真正差異來自資深 buyer、merchandiser、category manager 腦中的知識、runbooks 與歷史判斷。Skills 把這些知識變成 customer-facing systems 實際會使用的格式。

代表性 skills library

  • project-guidance：把模糊 query 轉成逐步施工計畫，包含依賴順序與常見錯誤。
  • materials-list：從 project description 產出 grouped bill-of-materials，補上常被忘記的品項。
  • review-summarize：將長篇 product reviews 摘要成 pros、cons 與常見用途。
  • delivery-window：依顧客位置、庫存與 freight network 計算配送選項。
  • return-policy：編碼退貨規則與例外；若要 action-allowed refund，需更嚴格 review。

Ownership：誰寫哪個 skill

Skill familyOwner teamWhy
project-guidance、product-fitTrades knowledge / category management這些團隊持有 merchandising 與 trade rules，且每週都會更新。
materials-listPro merchandisingPro segment 的 BOM logic 屬於 Pro team。
delivery-windowStore operations / fulfillmentreal-time inventory 與 freight rules 變動頻繁，應由處理它們的團隊擁有。
review-summarizePersonalization / data science最接近 NLP 與 customer feedback signals。
TierCapabilityReviewExamples
Read-Only可查詢或描述資料，不可改狀態Domain team approvalreview-summarize、store-locator、project-guidance
Draft-Only可產出供人 review 的內容，不可送出或提交Domain team + format ownerdraft-customer-email、materials-list
Action-Allowed可對真實系統執行不可逆操作Domain team + security/compliance + executive sign-offissue-refund、send-customer-message、reserve-inventory

競爭者可以取得相同 generic runtime，但不容易複製 company-specific skills library。投資 skills，即使用 generic runtime，也是在建立可持久的組織知識資產。
