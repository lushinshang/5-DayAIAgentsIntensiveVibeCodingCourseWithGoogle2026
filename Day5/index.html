引言

過去一年，Google 軟體工程師的日常例行工作有了完整的 180 度翻轉。在 2024 年初及之前，大量時間花費在深入研究開發者 API 與文件、手動逐行嘗試程式碼，以及判斷 Python 中應使用 substring in string、string.includes 或 string.contains 哪一種。程式碼完成後，需要投入相當力氣除錯，並解決功能程式碼與原始意圖之間的落差。如今，開發速度已達到超光速。

團隊現在使用 Antigravity [1] 或 Gemini CLI [2] 這類編碼代理程式，不僅能建議文字，還能實際使用工具並執行任務。AI 編碼編輯器能迅速輸出數千行文件完整的程式碼。彷彿僱用了一支永不休眠、永不抱怨的實習生大軍。但有個陷阱：雖然速度加快了，「速度的幻覺」也是真實存在的。雖然 bug 與程式碼的比率仍是挑戰——因為 AI 寫程式碼更快，也能以前所未有的速率產生潛在的錯誤。

由於實現不再是主要瓶頸，AI 可以被用來撰寫比任何人在相同時間範圍內更全面的測試覆蓋。這是提升程式碼信心的強大、程式化方式。AI 的潛能遠遠超越實現：它在系統設計、規格撰寫、路線圖規劃與結果分析上均表現出色。

「Vibe coding」指的是基於「感覺」或高層次意圖、而非嚴格手動編碼，使用 AI 快速生成程式碼。然而，至關重要的是澄清：「vibe coding」並非「生產環境中的 vibe」。雖然我們充分發揮 AI 代理程式的潛力，一切仍是完全有意圖且受控的，以確保生產級可靠性，而非仰賴未驗證的 AI 輸出。

在企業環境中，這個「感覺」被稱為 Agentic AI 開發。不同於標準生成式 AI（如同聰明的自動完成），Agentic AI 是混合型團隊成員；它使用語言模型來生成（大腦），並使用工具整合到工作流程中（雙手）。

規格驅動開發（SDD）

在傳統軟體工程世界中，開發者被教導要「程式碼優先」。一個模糊的想法會讓人開啟編輯器，不斷輸入直到有東西能運作。但在 Agentic AI 時代，日常工作發生了戲劇性的轉變。大部分時間現在花在撰寫高品質規格上——向 AI 確切說明應構建什麼的詳細技術指示。

在這個新工作流程中，開發者的角色更像技術架構師，而非傳統編碼者。關鍵在於：程式碼現在是可拋棄的。如果寫出了堅實的規格，整個程式碼庫可以反覆重新產生。代理程式甚至可以被指示在一個下午將整個專案從 Python 翻轉到 JavaScript。

優秀規格的特徵

優秀規格是建築北極星。它預防「脈絡碎片化」——數位版本的「傳話遊戲」，AI 因為查看檔案的過時快照而開始失去主線。AI 可以被用來精化優秀規格，本質上充當共同作者或規格編輯器/審閱者。通常，規格儲存在程式碼庫內，例如作為 specs/ 資料夾中的 Markdown 或 YAML 檔案。

一個用於產生新專案的優秀規格包含：

  • 完整技術設計：解決需求、資料庫模式和 API 規格。
  • 視覺輔助：包含圖表與具體工具及版本號碼的清單。
  • 背景資訊：給代理程式「為什麼」的背景於「什麼」。
  • 情境：好的樣子是什麼、什麼是錯的，並包含邊界案例。

應使用哪種格式？

2026 年由 Ouyang 等人進行的研究「SkCC：跨框架 LLM 代理程式的可移植安全技能編譯」[3]，揭示 LLM 代理程式對指示的格式化方式極度敏感，使用通用、未最佳化的 Markdown 檔案時性能最多下降 40%。對於使用 Gemini 的團隊，最佳格式化策略是混合 Markdown + 條件式 YAML 方法。資料顯示，對於深層嵌套的配置，YAML 達到 51.9% 的解析準確度，相較之下 JSON 只有 43.1%，XML 只有 33.8%。

行為驅動開發（BDD）

行為驅動開發（BDD）規格是將模糊、不明確的人類想法轉變為 AI 代理程式能夠不猜測而構建的精確架構設計的終極工具。BDD 規格使用一種稱為 Gherkin [4] 的標準化語法。Gherkin 依賴一個簡單的陳述式模板：Scenario / Given / When / Then。它強制 LLM 以「狀態 > 動作 > 結果」的方式思考。

提示

我經常在 Google Docs 中撰寫技術設計，讓架構計畫被許多人讀取與審查。審查完成後，使用「檔案 > 下載 > Markdown」並將該檔案加入工作區的 specs/ 資料夾。大型語言模型處理 token 化文字——每個字元消耗預算、時間和脈絡容量。透過將 /specs 資料夾視為精實、編譯指示集，消除推理「格式稅」。

指示應儲存在哪裡？

為實踐 SDD，需要理解編碼工具如何消耗指示。指示可以存在於不同的位置：

  • 1. 聊天介面（短暫、會話特定）：純粹用於高層次協調和即時反饋迴圈。
  • specs 資料夾（任務特定、簽入版本控制）：儲存技術設計、BDD 情境、API 契約與結構性 YAML 模式。./my-app/specs/my_spec.md
  • 2. 代理程式技能（可重用、功能/行為焦點）：結構化 Markdown 檔案，必須儲存在指定的 .agent 目錄中才能被 Antigravity 工作區管理器識別。
  • 3. 系統提示（全域、身份焦點）：Gemini CLI 和 Google Antigravity 掃描並按層次連接脈絡檔案，從全域覆蓋層層向下到本地專案配置。

不同使用案例的不同提示詞

將規格轉變為程式碼的方式不止一種，可分解成幾種執行模式：

  • 1. 專案產生（架構師）：從頭鷹架搭建，禁止 YOLO 模式。代理程式應先提議資料夾結構與技術堆疊以供確認，包含版本號碼。
  • 2. 功能產生（建築工人）：在現有程式碼庫之上實現功能，配合現有風格，多檔案編輯時手動確認差異。
  • 除錯（法醫專家）：從症狀提示轉換到證據提示，使用 gcloud logging read 等命令拉取記錄，先產生失敗的單元測試再修復。
  • 3. 文件撰寫（作者）：在 SDD 中文件是信實之源，使用 Google 風格 Docstrings（Python）或 JSDoc（TypeScript）。
  • 4. 資料工程（圖書館員）：使用 Google Cloud 資料擴充功能 [5]，提示詞代理程式始終顯示用於產生輸出的特定 SQL 查詢。

MCP：一次整合，普遍適用

MCP（模型情境協定）由 Anthropic 創建，現已成為開放標準。人們喜歡稱之為「AI 工具的 USB-C」。你為你的資料庫（或 API、檔案系統）建置一個 MCP 伺服器，任何相容 MCP 的代理程式都能使用它，不必撰寫客製化整合。

建置 MCP 伺服器

這是一個可運作的伺服器，約 40 行程式碼，公開一個 SQLite 資料庫：

Snippet 1 · Python
# mcp_server.py - Exposes a SQLite database via MCP
import sqlite3
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp.types import Tool, TextContent

server = Server("knowledge-base")
db = sqlite3.connect("knowledge.db")

@server.list_tools()
async def list_tools() -> list[Tool]:
    return [
        Tool(name="query_knowledge", description="Query the knowledge base with SQL",
             inputSchema={"type":"object","properties":{"sql":{"type":"string","description":"SQL query to execute (SELECT only)"}},"required":["sql"]}),
        Tool(name="add_knowledge", description="Add a new knowledge entry",
             inputSchema={"type":"object","properties":{"title":{"type":"string"},"content":{"type":"string"},"tags":{"type":"string","description":"Comma-separated tags"}},"required":["title","content"]}),
    ]

@server.call_tool()
async def call_tool(name: str, arguments: dict) -> list[TextContent]:
    if name == "query_knowledge":
        sql = arguments["sql"]
        if not sql.strip().upper().startswith("SELECT"):
            return [TextContent(type="text", text="Error: Only SELECT queries allowed")]
        cursor = db.execute(sql)
        rows = cursor.fetchall()
        columns = [desc[0] for desc in cursor.description]
        result = [dict(zip(columns, row)) for row in rows]
        return [TextContent(type="text", text=str(result))]
    elif name == "add_knowledge":
        db.execute("INSERT INTO knowledge (title, content, tags) VALUES (?, ?, ?)",
                   (arguments["title"], arguments["content"], arguments.get("tags", "")))
        db.commit()
        return [TextContent(type="text", text="Knowledge entry added.")]

async def main():
    async with stdio_server() as (read, write):
        init_options = server.create_initialization_options()
        await server.run(read, write, init_options)

if __name__ == "__main__":
    import asyncio
    asyncio.run(main())

Snippet 1: mcp_server.py：此程式碼片段使用 Python SDK 實現 MCP 伺服器，將 SQLite 資料庫公開為一組工具。

連結 MCP 用戶端

Snippet 2 · Python
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

server_params = StdioServerParameters(command="python", args=["mcp_server.py"])

async with stdio_client(server_params) as (read, write):
    async with ClientSession(read, write) as session:
        await session.initialize()
        tools = await session.list_tools()
        print(f"Available: {[t.name for t in tools.tools]}")
        result = await session.call_tool(
            "query_knowledge",
            {"sql": "SELECT * FROM knowledge WHERE tags LIKE '%agent%'"}
        )
        print(result.content[0].text)

Snippet 2: mcp_client.py：此範例展示 MCP 用戶端透過 stdio 連線到本地伺服器以探索並呼叫工具。

團隊文化與流程演進

使用現代編碼代理程式需要在思維和團隊文化上做出改變。在使用現代工具的同時堅守舊流程，就像在馬車上裝噴射引擎——這項技術無法直接套用於 20 年前的工作流程並期待它起飛。常見痛點：

  • 合併衝突：多個開發人員在一小時內在同一個檔案著陸。
  • 審查僵局：一個龐大的 PR 變成了俄羅斯套娃式的子 PR。
  • 脈絡碎片化：代理程式引用過時的快照，鑄造出呼叫不再存在的函式的程式碼。

程式碼審查

高速度整合的策略：

  • 捆綁摘要與風險評估：每個 PR 包含 AI 生成的變更快照、潛在破損點和風險評估。
  • 重新想像所有權：人工審查聚焦架構藍圖完整性，風格問題交由自動化工具（如 SKILLS.md）處理。
  • 「條件式 LGTM」：審查者批准 PR，條件是通過所有自動化測試；測試轉綠後自動合併。
  • 定義無責怪文化：將問題歸因於破裂的整合流程，而非個別開發人員。
  • 代理程式程式碼審查：建置技能自動執行程式碼審查，可與 Github Actions [6] 或 Gemini Code Assist [7] 整合。

以下是一個實用的程式碼審查技能範本（code-check.md）：

Snippet 3 · Markdown（Skill 範本）
Act as a Senior Software Engineer and Security Researcher.
Review the provided code for this Github PR or Diff using these strict criteria:
Use the command line to fetch the Github PR: `gh pr view <PR NUMBER>`

1. **Critical Vulnerabilities:** Check for hardcoded secrets (API keys), SQL injection, XSS, or broken authentication.
2. **Logic & Efficiency:** Identify "off-by-one" errors, infinite loops, or redundant API calls.
3. **Readability:** Suggest better naming conventions or breaking down "megafunctions" into smaller pieces.
4. **Edge Cases:** What happens if the input is null? What if the network fails?

Output Format:
- **Description:** What is this PR doing? Explain in details.
ISSUES:
- ⚠ **Critical:** (Stop-ship issues)
- ⚠️ **Warnings:** (Code smells or style issues)
- ✅ **Best Practices:** (Specific lines to refactor)
- 💡 **Quick Win:** (One sentence summary of the biggest improvement)
When there are no issues return: LGTM

Snippet 3: code-check.md：此技能定義了針對安全和邏輯準則的自動化程式碼審查結構化工作流程。

持續程式碼分析的三層光譜

部署監視儲存庫的代理程式時，三個層級在誰擁有執行時和誰撰寫審查準則上有所不同：

  • Tier 1 — 受管（Managed）
    例如 GitHub 上的 Gemini Code Assist [8] 或任何 SaaS PR 審查者。現成的路徑。無需撰寫提示詞，無需管理基礎設施；按座位付費。權衡：使用廠商的審查意見，通用審查者會錯過特定領域風險概況（HIPAA、PCI、內部 SDK）。
  • Tier 2 — 混合式（Hybrid）
    例如 GitHub Action [6] 觸發 Antigravity CLI [9]。撰寫自己的審查技能，提交到儲存庫，從 CI 動作觸發以非互動模式執行並將結果作為 PR 評論發佈。提示詞、模型選擇、沙箱化和審查準則屬於你。
  • Tier 3 — 自訂（Custom）
    例如 Gemini Enterprise Agent Engine [10] 上的 ADK [13] 代理程式。完全擁有的路徑，部署到 Agent Runtime 獲得耐久 Sessions 和 Memory Bank，並透過 A2A [11] 協調其他代理程式。此類架構的範例可在 Siemens 案例研究 [17] 中找到。權衡：你擁有評估、可觀測性、成本，以及待命輪換。

三個問題告訴你需要哪個層級：

  1. 你的審查準則有多具體？通用 → Tier 1；特定於團隊或儲存庫 → Tier 2 或 3。
  2. 代理程式需要在執行之間記住事情嗎？不 → Tier 1 或 2；是（程式碼庫記憶、跨 PR 脈絡）→ Tier 3。
  3. 如果出現問題，最壞的情況是什麼？一個嘈雜的評論 → 任何層級；一個合併的退化或洩露的祕密 → Tier 3。

Tier 3 大規模運作：圖形原生程式碼理解

自訂程式碼審查執行時（Tier 3）支援更繁重的元件——圖形資料庫保存程式碼結構、向量存儲用於語意檢索、子代理程式管道用於分解。在最大遺留系統現代化中浮現的模式是在知識圖形上建置代理程式，將程式碼、文件、票券和設計 PDF 攝入 Spanner Graph [12]，讓代理程式結合三種檢索模式：圖形走訪（GQL）、向量搜尋（ANN）與全文搜尋。將工作分解到 ADK [13] 子代理程式管道中——搜尋代理程式、影響代理程式、任務分解代理程式，最後才是編碼代理程式。

在百萬行程式碼庫上的生產試點已將等效的重構工作從兩週縮短到幾個小時。

永續性

我們正在看到一個新現象：批准疲勞。根據 Quantum Workplace 研究（由 CNBC 報告），頻繁 AI 使用者比非使用者更有可能經歷高度倦怠的機率高出 45% [14]。為了保護團隊，可以從持續監督轉移到結構化邊界：

  • 數位靜音時段：設定邊界，以便批准請求不會滲透到夜間和週末。
  • 代理程式洞察會議：舉辦每週會議，開發人員分享 AI 對應物確定的模式，將孤立的發現轉化為共用的組織知識。

零信任開發：建立安全防護網

當代理程式在沒有足夠護欄的情況下行動時，意外後果可能立即發生。在一次例行性的程式碼更新中，一個簡單的提示要求建立一個按鈕，觸發了出乎意料的連鎖反應。瀏覽器代理程式自主點擊了新按鈕，連接到一個沒有電子郵件保護措施的已棄用舊版代理程式，導致五十位同事收到充滿幻覺內容的虛假電子郵件。

這個事件凸顯了脈絡幻覺風險：當 AI 缺乏充分的資料時，它有時會填補空白，使用其脈絡中存在的任何字串，包括硬編碼電子郵件地址或 URL 等敏感資訊。護欄不是可選的；它們是讓有用的工具不變成不可預測工具的關鍵。

實施護欄

自主代理程式由 LLM 驅動，是概率性而非決定性的。將約束硬編碼到系統提示中是脆弱的，脈絡會溢出，代理程式可以透過提示詞注入被「說服」繞過規則。為了建立生產級平台，需要外部的、防篡改的治理機制。

沙箱化

除了淨化字串，真正的安全需要一個受限的執行環境，或「沙箱」，以約束代理程式的行動。透過在與主要網路和敏感檔案系統隔離的臨時、低權限容器內執行代理程式驅動的任務，建立一個「爆炸半徑」來保護核心基礎設施。

在 Antigravity 中，此安全防護網可以透過單個切換強制執行（終端沙箱化 [15]）。或者，透過撰寫自訂 Dockerfile（在 .gemini/sandbox.Dockerfile 中），從官方 Gemini CLI 沙箱 [16] 映像開始，並設定 export GEMINI_SANDBOX=docker 強制 CLI 工具完全在 Docker 中執行。

Human-in-the-Loop

高風險操作需要 Human-in-the-Loop（人工介入迴圈，HITL）協議作為最終故障保護。這涉及為符合特定風險概況的操作實施「檢查點」閘門，例如將程式碼部署到生產環境、修改資料庫結構描述或啟動金融交易。透過向人工主管提供代理程式淨化後的意圖以供手動簽核，AI 的速度與開發人員的細微判斷達到了平衡。

AI 生成測試覆蓋

AI 生成程式碼的激增帶來了從手動測試到 AI 生成測試覆蓋的轉變。在高速度環境中，「測試驅動開發」透過讓機器撰寫驗證其自身輸出的測試變成了現實。此過程涉及強制代理程式在嘗試任何修復之前產生失敗的單元測試或重現命令（如 curl 請求）。

評估

為什麼我們需要對使用 ML 模型的程式碼進行特殊品質檢查？傳統軟體測試對於輸出是生成而非計算的系統是不充分的。代理程式可以在其工具上通過 100 個單元測試，但仍然可能透過選擇錯誤的工具、改述關鍵答案或幻覺事實而慘敗。

評估透過用評分判斷和容許帶取代二進制斷言來彌補這一差距。評估詢問「代理程式的行為至少與基線一樣好嗎？」——來自 LLM-as-judge 的 0-5 分數、容許工具呼叫中排序變化的軌跡檢查。測試捕捉確定性退化；評估捕捉行為漂移。

政策伺服器

混合政策伺服器在操作觸及外部系統之前攔截它們。它在兩個層級上運作：

  • 結構性閘門（交通燈）：基於角色和環境的確定性規則。快速、二進制的檢查（例如，檢視者角色無法使用 send_email 工具）。此層防止架構違規，而無需詢問 LLM。
  • 語意閘門（智能裁判）：使用次要的、專門的 LLM（如 Gemini）來檢查建議操作的意圖和內容是否違反自然語言隱私指南。例如，管理員可以使用 send_email，但不應發送未遮罩的 PII（如純文字電子郵件地址或 API 金鑰）。

這些規則在標準的 policies.yaml 中定義：

Snippet 4 · YAML
environments:
  localhost:
    blocked_tools:
      - send_email
roles:
  viewer:
    allowed_tools:
      - list_files
      - read_file

Snippet 4: policies.yaml：此配置根據角色和環境建立了工具級權限的確定性閘門規則。

以下是政策伺服器的輕量級實作：

Snippet 5 · Python
import os, yaml
from google.genai import Client

class PolicyService:
    def is_tool_allowed(self, tool_name: str) -> bool:
        # Check Environment Blocks
        env_config = self.config.get("environments", {}).get(self.env, {})
        if tool_name in env_config.get("blocked_tools", []):
            return False
        # Check Role Permissions
        role_allows = self.config.get("roles", {}).get(self.role, {}).get("allowed_tools", [])
        return "*" in role_allows or tool_name in role_allows

    async def check_action_semantic(self, action_description: str) -> bool:
        client = Client(vertexai=True, project=self.project_id, location=self.location)
        prompt = f"Evaluate if this action violates PII policies: {action_description}"
        response = client.models.generate_content(model="gemini-3.1-pro", contents=prompt)
        return not response.text.strip().upper().startswith("VIOLATION")

Snippet 5: policy_server.py：此服務實作混合政策引擎以攔截操作並執行結構性和語意評估。

執行流程：當代理程式決定使用工具時：

  1. 結構性檢查：此角色/環境允許使用該工具嗎？（檢查 YAML）。
  2. 語意檢查：參數是否安全？（詢問 Gemini）。未遮罩的電子郵件地址被視為違規。
  3. 執行：如果兩者都通過，工具執行。否則，「政策違規」訊息返回給代理程式，允許其自我更正或優雅地失敗。

脈絡衛生 & 提示詞淨化

自主開發中的一個顯著危險是「脈絡幻覺」風險。當代理程式缺乏特定資料時，它可能會填補空白，使用其當前脈絡中的任何可用字串，可能洩露像硬編碼電子郵件地址或私人 URL 等敏感資訊。

為了降低此風險，透過執行 PII 遮罩和預留位置注入的中介軟體來實施嚴格的脈絡衛生。

實施動態脈絡解析器

此腳本充當核心轉換引擎，用執行時覆蓋或環境配置取代以雙方括號語法 [[VARIABLE_NAME]] 結構的預留位置字串：

Snippet 6 · Python
import os
import re
from typing import Optional, Dict, Any

def resolve_context(template_str: str, override_state: Optional[Dict[str, Any]] = None) -> str:
    """Scans a template string for [[VARIABLE_NAME]] and replaces it with values
    from override_state or os.environ."""
    if template_str is None:
        return ""
    state_to_check = override_state or {}

    def replacement(match):
        var_name = match.group(1).strip()
        # 1. Prioritize runtime state overrides
        if var_name in state_to_check and state_to_check[var_name] is not None:
            return str(state_to_check[var_name])
        # 2. Fallback to validated environment variables
        elif var_name in os.environ and os.environ[var_name] is not None:
            return os.environ[var_name]
        # 3. Leave unresolved to prevent silent failures
        else:
            return match.group(0)

    # Resolve all bracketed variables dynamically, e.g., [[COMMENTER_EMAIL]]
    return re.sub(r'\[\[([^\]]+)\]\]', replacement, template_str)

Snippet 6: context_resolver.py：此工具透過環境變數解析括號預留位置以維持脈絡衛生。

為了全域強制脈絡衛生，淨化工具必須直接接入代理程式的執行管道作為驗證步驟：

Snippet 7 · Python
# ... inside the validate_tool_call framework execution ...
resolved_args = {}
for k, v in args.items():
    if isinstance(v, str):
        resolved_args[k] = resolve_context(v)
    elif isinstance(v, list):
        resolved_args[k] = [resolve_context(i) if isinstance(i, str) else i for i in v]
    else:
        resolved_args[k] = v
args.clear()
args.update(resolved_args)

Snippet 7: tool_policy_engine.py：此中介軟體將脈絡解析器整合到代理程式管道中，以在執行前淨化工具參數。

透過強制此邊界，引擎將像 [[COMMENTER_EMAIL]] 或 [[DEFAULT_PRESENTATION_ID]] 這樣的通用預留位置安全且靜默地轉譯為授權測試資產，消除了在測試套件或系統提示中硬編碼敏感 PII 的需求。

摘要

在不到一年的時間裡，軟體開發週期變得極為迅速。看著代理程式在午餐前產生一千行文件完善的程式碼，是一項重大進步。

但那速度揭示了一個重要的轉變。AI 已經消除了程式碼生產瓶頸，將約束下移至必須審查、測試和整合那些產出的人類。這是共享的認知負荷：人類充當架構師，撰寫測試規格、整合規格和 MLOps/DevOps 藍圖，而 AI 執行繁重工作——撰寫實際測試程式碼、執行整合和管理細粒度的運營細節。

更好的提示詞和更快的模型單獨無法修復整合瓶頸。成功依賴於演進團隊動態、精化與代理程式的協作，以及為永不休眠的工具設定嚴格的邊界。挑戰已從純粹的程式碼生產，轉變為協調驗證、整合和部署工作的系統。
