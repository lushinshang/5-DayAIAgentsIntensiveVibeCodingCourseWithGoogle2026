你以為在用 AI 加速，其實在用 AI 借貸

85%。

這是 2026 年初，專業開發者每天使用 AI 編程工具的比例。再加上 41% 的新程式碼現在是 AI 直接生成的。這兩個數字並排放在一起，聽起來像是一則科技樂觀主義的廣告詞。但它們背後隱藏的問題，才是這堂課存在的真正理由。

大多數開發團隊都困在同一個地方：從「提示 AI」到「部署到生產環境」之間的那個峽谷。

你跑 prompt，AI 吐出程式碼，你貼上去跑看看，壞掉了，把錯誤訊息貼回去叫 AI 修，再跑，再壞，再修——這個迴圈不是在建設，而是在累積。每一次「感覺好像可以了」都是一張支票，遲早要兌現。而當帳單來的時候，你甚至不知道問題在哪裡，因為你從頭到尾都沒有真正理解那段程式碼在做什麼。

這種工作模式有個名字，叫做 Vibe Coding。

📌 概念定義 Vibe Coding 是一種完全順著「感覺走」的 AI 輔助開發方式：描述需求、接受輸出、遇錯即貼、循環迭代。驗證標準是「跑起來了嗎」，而不是「行為正確嗎」。
💡 台灣電商情境類比 Vibe Coding 像是到夜市擺攤，生意好的時候什麼都快，但颱風天一來（流量暴增或需求驟變），你才發現攤車根本沒有固定好。Agentic Engineering 則是開正式餐廳：前期投入裝潢、衛生認證、SOP，但撐得過每一個週末。
⚠️ 最常見的誤解 「我有在用 AI 寫程式，所以我在做 Agentic Engineering。」這是最危險的認知錯誤。用了什麼工具不重要，如何驗證輸出才是分水嶺。

這不是光譜兩端，而是兩種截然不同的工程哲學

「從 Vibe Coding 到 Agentic Engineering 是一條光譜」——這個說法本身就值得懷疑。光譜的比喻暗示你可以站在中間某個位置，舒適地「半嚴謹半隨意」地工作。但實際上，這兩者之間的差異不是程度問題，而是結構性問題。

Vibe Coding 的核心預設是：AI 的輸出大概是對的，有問題再說。

Agentic Engineering 的核心預設是：AI 的輸出必須被驗證，驗證機制必須是自動的、系統性的、可重現的。

具體來說，Agentic Engineering 要求你建立：

  • 正式規格（formal specifications）：你必須能說清楚「正確」長什麼樣
  • 自動化測試套件（automated test suites）：機器驗，不是眼睛驗
  • 持續整合閘門（CI/CD gates）：每一次更動都要過關才能繼續

這三樣東西加在一起，讓 AI 的輸出進入一個有邊界的沙盒，而不是直接落地到生產環境。

🧪 思考驗證 你上次部署的功能，你能說出「它正確的行為標準」是什麼嗎？如果不能，你現在是在哪個位置？

公式背後藏著一個讓人不舒服的真相

如果只能記住一個公式，記這個：

Agent = Model + Harness
模型佔 10%　　Harness 佔 90%

這個數字讓很多人第一次聽到會愣住。大家花了那麼多時間選模型、比較參數、測試哪個 LLM 更聰明，結果模型只佔了整個 Agent 系統有效性的一成？

答案是肯定的。因為讓 Agent 在生產環境中可靠運作的，是 Harness——那個包裹在模型外面的系統骨架。

📌 概念定義 Harness（圍套） 是 Agent 系統中除了模型之外的所有東西：沙盒環境（sandbox）、可呼叫的工具（tools）、任務協作邏輯（orchestration）、以及防止 Agent 做壞事的護欄（guardrails）。
💡 外科手術室類比 模型是外科醫師的判斷力，Harness 是整個手術室。再厲害的外科醫師，沒有無菌環境、手術器械、麻醉師、護理師，就什麼都做不了。你不會只評估醫師的智商來決定要不要動手術。
⚠️ 常見誤解 「換一個更好的模型，Agent 就會更可靠。」這個思路是把 90% 的問題用 10% 的解法去解。在大多數 Agent 失敗的案例裡，問題出在 Harness 設計不良，而不是模型不夠聰明。

你以為在學 Prompt，其實在學 Context

課程把 2026 年工程師最重要的技能定義為 Context Engineering，而不是 Prompt Engineering。

這兩個詞的差距，比看起來大得多。

Prompt Engineering 問的是：「怎麼問 AI 才能得到好答案？」

Context Engineering 問的是：「AI 需要知道什麼、以什麼形式知道、在什麼時機知道，才能在整個任務過程中持續做出好的判斷？」

前者是一次性的；後者是架構性的。

在 Harness 的設計裡，Context 分成兩大類：

  • 靜態 Context（如 system instructions）：每次都帶著，貴但穩定
  • 動態 Context（如 Agent Skills 按需載入）：用到才帶，便宜且精準

這個區別直接影響系統成本。一個設計不良的 Agent 會把整個 codebase 都塞進 context window；一個設計良好的 Agent 只帶當下任務需要的部分。

最後那 20%，才是真正的戰場

課程裡有一個反覆出現的概念：Last Mile Problem。

AI 可以在幾秒鐘內完成一個功能的初稿——80% 的工作量消失了。但剩下的那 20%，才是讓系統能否在真實生產環境中運作的關鍵。那 20% 包括：邊緣案例處理、一致性、錯誤恢復、效能在壓力下的表現。

這 20% 不會因為你的模型更好而自動消失。

真正能跨越最後 20% 的，是一套完整的迴圈：

Scaffold → Build → Observe → Optimize

更具體地說：

  1. 能寫程式的 Agent：讓 AI 不只是呼叫 API，而是在沙盒裡自己產生工具、產生子 Agent
  2. 自動驗證迴圈：子 Agent 評估主 Agent 的輸出，自動迭代
  3. Human-in-the-loop：設定條件，當信心不足時觸發人工審核
  4. 黃金資料集的累積：把每一次人工審核的結果存起來，成為自我改善的訓練材料
📌 概念定義 Scaffold（鷹架） 是 Agent 系統啟動前預先建立的結構：執行環境、工具定義、任務邊界。它讓 Agent 知道「我在一個有規則的房間裡工作」，而不是「我可以做任何事」。
🧪 思考驗證 你的 Agent 系統有沒有辦法在沒有人工介入的情況下，自己發現自己做錯了？如果沒有，你現在跨越最後 20% 的方法是什麼？

一個沒有人想正視的風險

課程的 Q&A 裡有一段討論，很少人會在 AI 樂觀主義的浪潮裡認真對待它。問題是：AI 驅動的 SDLC 有哪些長期風險？

給出的答案分三層，層層遞進，越往後越不舒服：

風險一

程式碼專業知識的侵蝕。當 AI 越來越成功地接管了寫 code、測試、修 bug 的工作，開發者對程式碼庫的深度理解會慢慢消失。你不再親手寫那些邏輯，你就不再真的懂它們。

風險二

問責的消失。當問題發生時，誰負責？人寫的 bug，有人負責。AI 生成的 bug，而你沒辦法說清楚那段程式碼在做什麼，問責的鏈條就斷了。

風險三

創新機會的流失。工程師的很多突破性想法，來自「深入摸索一個系統到底在做什麼」的過程中。如果你從來不需要深入，你也永遠不會在深入的過程中發現新的可能性。

⚠️ 重要提醒 這三個風險不是遙遠的未來式，而是已經在某些團隊發生的現在式。真正的防禦策略不是拒絕 AI，而是刻意設計「保留人類深度理解」的工作流程。

自我進化的 Agent，以及你的想像力邊界

課程裡提到一個叫做 Alpha Evolve 的系統，值得單獨拿出來說。

它不是一個普通的 Agent。它是一個「進化型演算法 Agent」：給定一個評估函數（evaluator function），它會自動尋找、測試、改進演算法，直到找到比人類已知最佳解更好的方案。

它在 Google 內部用於：

  • 數學問題（矩陣乘法的新優化，解決了數十年沒有突破的問題）
  • YouTube 的 TensorFlow → JAX 框架遷移（速度快了 6 到 8 倍）
  • 硬體架構設計
  • 分子模擬、DNA 定序

這個系統告訴我們什麼？AI Agents 可以應用的範圍，上限是你願意定義「成功」的能力，而不是 AI 的能力。

你能不能寫出一個評估函數，說清楚「什麼叫做更好的結果」？如果可以，Alpha Evolve 就能幫你找到更好的解法。如果不能，那個瓶頸在你，不在 AI。

長跑 Agent 的真實瓶頸在哪裡

Deep Research 是目前最廣為人知的長跑 Agent 成功案例之一；AI 編程 Agent 是另一個。這兩個用例為什麼成功？

Deep Research 成功是因為：答案本身需要大量的背景脈絡鋪墊，長跑才能積累足夠的脈絡。

AI 編程 Agent 成功是因為：可以在運行過程中不斷跑測試，每一個增量動作都能即時驗證。驗證機制把長跑的風險壓低了。

這裡有一個有趣的深層洞察：現有的工具都是為人類設計的，而不是為 Agent 設計的。API、外部服務、資料庫存取——它們假設使用者有一定的延遲容忍度，假設操作是序列性的，假設會有人在另一端等待回應。但 Agent 不等待，Agent 要並行，Agent 要毫秒級反應。

這意味著，隨著 Agent 能力越來越強，我們會看到一波「為 Agent 重新設計的工具」浪潮。這個方向，可能是接下來幾年最值得關注的工程機會之一。

IUS：一個比「做得到」更重要的問題

最後談一個框架，簡單到讓人低估它，但實際上是課程裡最實用的思考工具。

I
Impressive
令人驚豔的 demo，成功展示可能性
U
Useful
能普遍複製，對他人也有效
S
Sustainable
成本、安全、合規皆可長期維持

大多數人做 AI 應用，卡在 I 和 U 之間。你做出了一個漂亮的 demo，但 demo 依賴的那些條件，無法複製到別人的環境裡。

而從 U 到 S 更難。有些 AI 解決方案，效果比現有方法好，但成本是現有方法的三倍。這不叫「AI 的勝利」，這叫「AI 在燒錢」。

Sustainable 要求你問：這個解法，在成本、安全、法規、可擴展性上，能撐過五年嗎？

課程裡舉的例子很直白：如果你的解法比原來貴三倍，但公司不可能永遠補貼這個成本，那你做的只是一個暫時性的展示，不是一個真正的解法。

你真正要帶走的，不是技術，是判斷力

整堂課的核心訊息，壓縮成一句話：

實作的能力，已經不再是瓶頸。
規格的品質，才是新的瓶頸。

當 41% 的程式碼都是 AI 生成的，「你會不會寫程式」變成了一個越來越不重要的問題。真正重要的問題是：

  • 你能不能說清楚你想要什麼？（規格）
  • 你能不能驗證你得到的是你想要的？（評估）
  • 你能不能設計一個系統，讓 AI 在邊界內可靠地工作？（Harness）

這三件事，沒有一件可以交給 AI 做。它們是人類在這個新 SDLC 裡的核心工作。

教育的未來不是「學更多工具」，而是「從大學就開始帶著整個產品出來」。證明你能把想法轉化成可用的系統，這個能力，比任何語法知識更難偽造，也更難替代。

🧪 最後一個思考驗證 你上一個 AI 輔助的專案，它是 Impressive、Useful，還是 Sustainable？差了哪一步？那一步，你現在有辦法補上嗎？
