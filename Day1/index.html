前言

在電腦科學歷史的大部分時間裡，程式設計一直是一種「翻譯」的行為：用人類的語言理解問題、用抽象的概念設計解決方案，然後用機器可以執行的語法將其呈現出來。每個步驟都會帶來摩擦。而現在，這種摩擦正在消逝。軟體工程正經歷著自高階程式語言問世以來最重大的變革。

數十年來，開發者與機器之間的主要介面一直是「語法」：大括號、分號、型別註解，以及程式語言的精準文法。那個時代即將結束。

一個新的範式已經到來，開發者只需表達他們想建構「什麼」，而不是「如何」建構。機器負責實作，人類則提供意圖、架構和判斷。這不是遙遠的未來——這是快速增長的專業開發者每天的真實寫照。截至 2026 年初，有 85% 的專業開發者定期使用 AI Coding Agents，51% 每天使用，且估計所有新程式碼中有 41% 是由 AI 生成的。

這種轉變並非一夕之間發生。它始於 Autocomplete（編輯器中簡單的 token 預測）；接著是能完成整個函式的行內程式碼建議（Inline Code Suggestions）；隨後，基於對話的介面讓開發者能用自然語言描述功能並取得可運作的實作；而現在，完全自主的 Agents 可以複製 Repo、規劃跨檔案修改、在沙箱環境中執行、跑測試，最後提交 PR——整個過程不需要人類輸入任何一行程式碼。

每一代技術在保留前代成果的同時，也不斷拉高單一工程師所能達到的成就上限。

這對軟體開發生命週期（SDLC）的影響極為深遠。從需求收集、部署到維護，每一個階段都正在被 AI 能力重塑。然而，這種轉變並非單一且簡單的。其光譜的一端是隨性的「Vibe Coding」（開發者給 AI 下 prompt 並直接接受任何回傳的結果），另一端則是嚴謹的「Agentic Engineering」（AI 在精心設計的限制條件、測試和回饋循環系統中，作為強大的實作引擎，而人類仍保留對架構、正確性與品質的監督權）。

這兩者的區別至關重要。跟 CTO 說你的團隊正在對金流處理系統進行 Vibe Coding，這將會（且理應）敲響警鐘；但如果跟同一位 CTO 說你的團隊正在實踐 Agentic Engineering，由 AI 在人類設計的限制條件下處理實作，並透過測試覆蓋率確保正確性，這本質上是完全不同的對話。

本報告為這場對話奠定了基礎。我們將追溯從隨性的 Vibe Coding 到嚴謹的 Agentic Engineering 的光譜，檢視開發者的角色如何從「撰寫程式碼」轉變為「行使判斷力」——從指揮者（conductor）到編排者（orchestrator）——並說明如何採用這些工具，以建構出真正值得信賴的軟體。

為什麼是這篇報告，為什麼是現在

新工具、新功能與新範式每週都在湧現。工程團隊需要一個架構來理清這個領域的脈絡——這不應該是一個幾個月後就會過時的斷代紀錄，而是一套即使在特定工具演進時依然適用的原則與心智模型。

這篇報告適合的讀者群

本報告適合軟體工程師、工程經理、架構師以及技術領導者，幫助他們理解 AI 如何重塑 SDLC，並在不犧牲 Production 軟體所需嚴謹度的前提下，採用這些新能力。我們假設讀者熟悉現代軟體開發實務，但不需要具備 AI 或機器學習的專業細節。

從語法到意圖的轉變

在深入探討之前，我們需要對「Agent 是什麼」以及「Vibe Coding 到底代表什麼」建立共識。這兩個詞彙都已經累積了相當多的含意，需要仔細釐清。

AI Agents：快速複習

AI Agent 是一個軟體系統，它能感知目標、規劃達成步驟、透過工具採取行動、觀察結果，並不斷迭代，直到達成目標或達到停止條件。相較於 Chatbot 產生回應後就等待下一個 Prompt，Agent 則會執行自己的循環。您在最頂層給它一個目標，接著它會在每個步驟中自行決定下一步該做什麼。

不論簡單或複雜，每個 Agent 都由五個部分組成。2025 年 11 月發表的《Introduction to Agents》白皮書對各個部分進行了深入探討，在此我們提供一個精簡版本：

  • Model 是推理引擎。它讀取當前的 Context，決定下一步該怎麼做，並產生下一個想法、下一次工具呼叫或下一條訊息。
  • Tools 將 Model 與外部世界連結。它們包括 Agent 可以呼叫的 API、可以執行的程式碼、可以查詢的資料庫，以及可以委派的其他 Agents。
  • Memory 是狀態。它使 Agent 能夠回想過去的互動、檢索特定專案的規則，並跨 Session 保留 Context，使其永遠不會從空白狀態開始。
  • Orchestration 是執行該循環的程式碼。它為每次模型呼叫組合 Context、分派工具呼叫、擷取其結果，並決定是否繼續執行。
  • Deployment 是將 Prototype 轉化為服務的關鍵：代管（hosting）、身分識別、可觀測性，以及 Agent 運作所依賴的 Production 基礎設施。

這四個部分在一個持續的循環中協同工作：獲取任務、掃描場景、深思熟慮、採取行動、觀察並迭代。這個循環是每個 Agent 的核心心跳。本報告中的其他內容，以及課程其餘部分的所有內容，都是這個循環的變體。

什麼是 Vibe Coding？

在 2025 年 2 月，Andrej Karpathy 發布了一篇關於新程式設計方式的描述，在軟體工程社群中引起了廣泛共鳴。他描述了一種「完全沉浸在 Vibe 中、擁抱指數級成長，甚至忘記程式碼存在」的方法。

在這種模式下，開發者使用自然語言描述他們想要的東西、接受 AI 的輸出，而當某些地方出錯時，就將錯誤訊息複製回 Prompt 中，要求 AI 進行修復。

這個詞彙迅速走紅，因為它捕捉到了真實的現象：許多開發者已經在用這種方式工作，只是苦於沒有合適的詞彙來形容。在短短幾個月內，「Vibe Coding」成為任何 AI 輔助開發工作流的通稱，這也引發了混淆。資深工程師使用 AI 助理來實作規格明確的功能，算是「Vibe Coding」嗎？團隊使用 AI Agents 來執行精心規劃的架構，也算嗎？這個詞彙被應用得如此廣泛，以至於開始失去其原本的具體意涵。

到了 2026 年初，Karpathy 本人也承認最初的框架過於狹隘，因而引入了「Agentic Engineering」這個詞彙，用以描述光譜中更為嚴謹的另一端。

光譜：從 Vibe Coding 到 Agentic Engineering

與其將 Vibe Coding 和 Agentic Engineering 視為非黑即白的二分法，我們認為將它們視為一個光譜的兩端會更有幫助。關鍵的區別並不在於你是否使用 AI，而是在於圍繞 AI 產出的結構化程度、驗證機制以及人類判斷的多寡。

實務技巧

在這個光譜上的正確位置取決於賭注有多大。週末做個原型專案可以純粹是 Vibe Coding；而處理金融交易的生產環境 API 則需要 Agentic Engineering。大多數的實際工作都落在兩者之間，而開發者的技能就在於知道如何為每項任務劃定界線。

光譜兩端之間單一最大的區別在於「產出如何被驗證」。在 Vibe Coding 中，驗證是選配的；開發者執行程式碼並檢查它看起來是否正確。在 Agentic Engineering 中，則是由兩種機制協同工作。測試（Tests）用來驗證系統中具確定性的部分：給定某個輸入，函式就會產生對應的輸出。而評估（Evaluations，或稱 Evals）則用來驗證不具確定性的部分：Agent 是否採取了正確的步驟軌跡、選擇了正確的工具，並產生了符合品質標準的最終回應。測試是由程式碼來檢驗；Evals 則是由標記資料集（labelled datasets）、評分量表（scoring rubrics）和 LM 裁判（judges）來檢驗。如果沒有這兩者，無論你的 Prompting 多麼複雜，這種實踐本質上依然只是 Vibe Coding。

Context Engineering：真正的核心技能

隨著這個領域的成熟，出現了一個關鍵的洞察：AI 生成程式碼的品質，與其說取決於你 Prompting 的巧妙程度，不如說更取決於所提供 Context 的品質。這個體悟催生了 Context Engineering 的概念，亦即向 AI Agent 提供關於程式碼庫、架構、規範和意圖的豐富且結構化資訊的實踐。

開發者必須考慮六種主要類型的 Context：

  • 指令（Instructions）：Agent 的核心角色、目標和運作邊界。
  • 知識（Knowledge）：檢索出的文件、架構圖和特定領域的資料。
  • 記憶（Memory）：短期的 Session 日誌（剛剛發生了什麼）以及長期的持久狀態（這個專案是什麼）。
  • 範例（Examples）：Few-shot 的行為展示和程式碼庫參考模式。
  • 工具（Tools）：Agent 可以呼叫的 API、腳本和外部服務的精確定義。
  • 護欄（Guardrails）：硬性限制、格式規則和安全性驗證。

在 AI 程式碼生成中，Context Engineering 涉及仔細平衡這六個元素中，哪些是 Agent 預先擁有的，哪些是它可以根據需求（on demand）檢索的。這在靜態 Context 與動態 Context 之間建立了關鍵的區隔。

靜態 Context（Static context）是始終載入的：系統指令、規則檔案（AGENTS.md、CLAUDE.md、GEMINI.md）、全域記憶和角色（persona）定義。它定義了 Agent 是誰以及它如何運作。靜態 Context 的成本很高，因為無論是否相關，每個 Token 都會存在於每一次互動中。

動態 Context（Dynamic context）是根據需求載入的：由任務比對觸發的技能指令、在執行期間檢索的工具結果、從 RAG 管線中獲取的文件，以及視窗化的 Session 歷史記錄。動態 Context 非常有效率，因為 Agent 只有在需要該資訊時才需要支付 Token 成本。

關於哪些內容屬於靜態 Context，哪些屬於動態 Context 的設計決策，是一個真正的工程權衡。太多的靜態 Context 會浪費 Token 並稀釋關鍵訊號。太少則意味著 Agent 會遺忘關鍵規則。最優秀的系統會將這個邊界視為一等架構決策（first-class architectural decision），並像其他任何設定一樣進行審查和版本控制。

管理動態 Context 最強大的模式是「Agent Skills」（Agent 技能）：這是結構化、可移植的程序性知識套件，Agent 只有在任務需要時才會載入。

與其將每一項專業知識都嵌入到 Agent 的系統 Prompt 中，Skills 允許 Agent 保持輕量化的通用型角色，並透過漸進式揭露（progressive disclosure）根據需求彈性地轉變為專家角色。Agent 在啟動時只會看到輕量化的詮釋資料（metadata），在任務比對成功時載入完整指令，並僅在明確需要時才拉取深入的參考資料。其結果是，一個 Agent 可以攜帶數十種專業能力，但只為其正在使用的一種支付 Token 成本。

Agent Skills 在各大 Coding Agent 和 enterprise 平台中得到了快速採用，因為它們解決了困擾 AI Agent 開發的四個問題：

  • 由於 Prompt 過載而導致的 Context 衰退（Context rot）
  • LLM 缺乏程序性記憶（procedural memory）的問題
  • 多 Agent 架構的營運開銷（operational overhead）
  • 跨工具和跨廠商的可移植性需求

本節介紹了 Context Engineering 的核心原則：每個 Agent 都需要的六種 Context 類型、靜態與動態 Context 之間的權衡，以及作為大規模管理該權衡之關鍵模式的 Agent Skills。

本系列中有關 Context Engineering 的第三天配套論文《Context Engineering: Sessions, Skills & Memory》將進一步探討這些概念，內容涵蓋如何設計和管理 Session、如何撰寫與評估 Skills、如何在不同互動之間建立持久記憶，以及如何為生產環境系統最佳化 Token 經濟學。

從「Prompt Engineering」到「Context Engineering」的轉變反映了與 AI 協同工作的一個更深層的事實。模型與其說需要巧妙措辭的指令，不如說更需要優秀的人類開發者要做好工作所需的相同 Context。問題並不在於「我該如何哄騙 AI 寫出好程式碼？」，而是「一個新的團隊成員需要知道些什麼才能有效率地做出貢獻，而我該如何將這些知識編碼成 AI 可以使用的形式？」

Context Engineering 是連接 Vibe Coding 和 Agentic Engineering 的橋樑。它也是連接本節與下一節的橋樑，在下一節中，我們將研究圍繞每個模型並使其發揮作用的結構。

透過將我們的焦點從撰寫語法轉移到對這個 Context 的設計（engineering），軟體開發中的瓶頸發生了根本性的變化。我們不再等待人類雙手敲鍵盤來寫樣板程式碼；我們是在等待人類大腦來定義邊界。這使得我們必須對傳統的軟體開發生命週期（SDLC）進行徹底的重新想像，因為我們用來建構軟體的系統現在主導了軟體交付的速度。

全新的軟體開發生命週期（SDLC）

傳統 SDLC 面臨的壓力

軟體開發生命週期已經歷過一次重大的變革。在過去的二十年中，大多數企業從循序漸進的瀑布式流程轉向迭代模型：Agile Sprint、持續整合（CI）、DevOps 管線和快速發佈週期。這一轉變縮短了回饋循環，使測試更接近開發，並使部署成為一個持續的過程，而不是每季一次的大事。

AI 極大地壓縮了這個週期，但壓縮的幅度並不均勻：以前需要數週才能完成的實作（implementation）現在幾小時內就能完成，而需求分析、架構設計和驗證則依然固執地維持著人類的速度。其結果並不是舊版 SDLC 的加速版。這是一個截然不同的工作流程，各個階段之間的邊界變得模糊，迭代週期從數週縮短到數分鐘，且開發者的角色也從主要的實作者轉變為系統設計師與品質仲裁者。

關於變革速度的說明：上述按階段呈現的圖景反映了截至 2026 年中 AI 驅動的 SDLC 狀態。它變化得非常快。早期跡象表明，壓縮效應將蔓延到實作之外：團隊已經在嘗試開發者直接從規格書跳到審查的工作流程，而 AI Agent 則在背景處理實作、測試和部署。本節中劃定的邊界在 12 個月後可能看起來會有所不同。但隨著機器承擔更多實作工作，人類的判斷力、品味以及驗證 AI 產出的技能將維持不變。

AI 如何轉變各個階段

需求與規劃

需求分析是意圖與實作之間落差歷史上最大的階段。將商業需求轉譯為技術規格書一直是一項繁瑣且容易出錯的手動過程，這會在利害關係人想要的與工程師建構的之間，產生持續的落差。

現代 AI 工具可以直接參與需求細化：從產品簡報中生成使用者故事、識別人類遺漏的邊緣案例（edge cases）、從自然語言描述中產生 API Schema，以及從規格書中生成互動式原型。Agentic 開發環境使開發者能夠在數分鐘內從描述轉化為可運行的原型，將從需求到原型的回饋循環壓縮至接近零。

需求不再是團隊之間交接的文件。它們變成了人類與 AI 之間的對話，同時產生規格與初始實作。

設計與架構

架構設計依然是 SDLC 中最固執地以人類為中心的第一階段，這是有充分理由的。架構決策本質上是關於權衡：一致性 vs. 可用性、複雜度 vs. 彈性、自建 vs. 購買。這些權衡取決於商業背景、組織限制以及 AI 無法完全理解的長期策略考量。

一旦做出架構決策，AI 在實作這些決策方面表現出色。給定一份清晰的架構文件，AI Agent 可以建立整個應用程式的起手架（scaffold）、在各個模組中生成一致的模式，並確保新程式碼符合既定的規範。開發者的角色從撰寫樣板程式碼轉變為做出並記錄這些樣板程式碼所實現的結構性決策。

實作

現代 Coding Agent 可以從自然語言描述中生成整個功能模組、實作複雜的演算法，並產生能協同工作的多檔案變更。生產力的提升是真實存在的：產業調查報告顯示有 25% 到 39% 的生產力提升，有些任務甚至能看到更顯著的成長。

然而，實際情況比這些標題數字更為細緻。METR 的一項研究發現，經驗豐富的開發者在使用 AI 輔助工具時，在某些任務上實際上多花了 19% 的時間，這主要是因為花在驗證、偵錯（debugging）和修正 AI 產出上的時間。AI 並沒有消除實作工作，而是將其從撰寫轉變為審查、引導和驗證。

測試與品質保證

測試 AI 生成的程式碼不僅需要評估 Agent 產出了什麼，還需要評估它是如何做到的。產出評估（Output evaluation）檢查最終的產出物（artifact）：程式碼是否能編譯？測試是否通過？軌跡評估（Trajectory evaluation）則檢查工具呼叫與中間推理的完整序列。兩者皆為必要，因為一個跳過驗證步驟但看似流暢的產出，其失敗的危險性遠高於一個帶有明顯錯誤的產出。

AI 同時也變革了測試生成本身。Agent 可以產生人類可能想不到的測試案例，包括邊緣案例和基於屬性的測試（property-based tests）。更重要的是，測試與 Evals 變成了向 AI Agent 傳達意圖的主要機制：一個寫得很好的 Eval 套件能告訴 AI 什麼是「正確」，並提供一種自動化驗證的方法。

當這些實踐被整合進持續品質飛輪（continuous quality flywheel）時最為有效：針對 Benchmark 套件進行評估、透過將根本原因分群來診斷失敗、最佳化導致問題的 Prompt 或工具、針對迴歸測試套件（regression suite）驗證修復，並監控生產環境流量以發現新的失敗模式。每一次循環都會帶來加乘效果。

程式碼審查與部署

審查流程本身也正在得到增強，AI 作為第一輪審查者，可以在人類審查者看到程式碼之前，識別出潛在的 Bug、樣式違規、安全性漏洞和效能問題。這並不能取代人類審查，因為有關設計、可維護性和策略一致性等依賴 Context 的決策依然需要人類判斷，但它顯著減輕了審查者的認知負擔。

部署管線（Deployment pipelines）也開始具備 AI 感知能力。AI Agent 可以監控部署健康狀況、自動回滾（roll back）有問題的發佈版本，並根據變更的性質與範疇來預測部署風險。現代部署平台正越來越多地與 AI 驅動的可觀測性（observability）相結合，以在生產環境行為與開發決策之間建立回饋循環。

本系列的第五天內容涵蓋了當 PR 數量隨著 Agent 的產出而大幅增加時，人類審查者所面臨的轉變——例如打包摘要（bundled summaries）、條件式 LGTM 以及 Agent 驅動的程式碼審查技能。

維護與演進

昔日對於新團隊成員而言高深莫測的舊有程式碼庫（legacy codebases），現在可以在 AI 的協助下進行導覽、理解與修改。AI Agent 可以閱讀程式碼庫、理解其模式、識別出變更所需的核心檔案，並在尊重既有架構的前提下實作修改。

這對技術債（technical debt）有著重大的影響。過去因為只有原作者理解而被視為「碰了太危險」的程式碼，現在可以被安全地重構（refactored）、現代化和擴充。AI Agent 可以系統化地在框架之間遷移程式碼庫、更新已廢棄的 API，並讓測試套件現代化——這些任務在過去是如此地繁瑣且危險，以致於根本不會發生。

工廠模型：建構用來開發軟體的系統

連結這些轉變的心智模型，就是我們所說的「工廠模型」（factory model）。在此模型中，開發者的主要產出不是程式碼，而是產生程式碼的系統。這個系統包含：

  • 定義需要建構內容的規格（Specifications）與 context
  • 將規格轉換為實作的 Agent
  • 驗證正確性的測試與品質把關機制（quality gates）
  • 將失敗結果回傳給 Agent 進行修正的回饋迴圈（feedback loops）
  • 限制 Agent 只能做出安全且可預測行為的防護欄（guardrails）

工廠經理不會親手組裝每一個小零件。他們設計裝配線並確保品質管制。現代開發者則是設計開發系統，並確保其產出符合要求的標準。成功關鍵在於給予 Agent 成功準則，而非一步一步的指令，然後讓它們進行迭代。

這引出了一個驅動本文後半段的問題：工廠裡的核心機器是什麼？在裝配線內實際工作的 Agent 本身，到底長得像什麼樣子？

Harness Engineering：圍繞在模型周圍的機制

當開發人員開始使用 AI agent 時，很容易會把模型當作系統本身。新模型推出時，Agent 就變聰明了；舊模型下，Agent 就變笨了。模型成了所有好壞表現的唯一解釋。

這種直覺是錯誤的，且會導致錯誤的投資。模型只是運行中 Agent 的其中一個輸入。其他所有部分——Prompting、工具、context 策略、hooks、沙箱（sandboxes）、子 Agent（sub-agents）、可觀測性（observability）——都是 harness：也就是包裹在模型周圍的起手架，讓它能夠真正完成任務。

原始模型並非 Agent。只有當 harness 賦予它狀態、工具執行、回饋迴圈和強制約束力時，它才會成為 Agent。開發者在與 Claude Code、Cursor、Codex、Antigravity、Aider 或 Cline 協作時所體驗到的行為，主要取決於 harness 的運作，而不單純是底層使用了哪種模型。

Harness 包含哪些內容

具體而言，harness 包含：

  • 指示與規則檔案（Instructions and Rule Files）：定義 Agent 是誰、它關注什麼、以及禁止做什麼的文字。這包括 AGENTS.md、CLAUDE.md、GEMINI.md、skill 檔案和子 Agent 的提示詞。
  • 工具（Tools）：Agent 可以呼叫的函式、MCP 伺服器和 API，以及周圍用來告訴模型何時以及如何呼叫它們的說明。
  • 沙箱與執行環境（Sandboxes and execution environments）：Agent 程式碼實際運作的地方、它可以存取什麼、以及無法觸及什麼。
  • 協調邏輯（Orchestration logic）：子 Agent 的生成、模型路由、專家之間的任務交接，以及控管各個組件何時啟動的規則。
  • 防護欄或 Hooks（Guardrails or Hooks）：在特定生命週期節點運行的確定性（deterministic）程式碼，例如在呼叫工具前、檔案編輯後、或是提交（commit）前。Hooks 是用來放置 Agent 不該忘記卻經常忘記的事情的好地方。
  • 可觀測性（Observability）：日誌、追蹤、評估、成本與延遲計量。若沒有可觀測性，就無法得知 Agent 是表現優異，還是在悄悄偏離軌道。

如果這聽起來涵蓋範圍很廣，確實如此。這是開發團隊必須負責的範圍，而不是模型提供商的職責。

SDLC 中的 Harness

雖然模型本身決定了如何完成任務，但 harness 是提供存取工具、沙箱以及執行任務所需之協調邏輯的起手架。因此，在 AI Agent 運行的每個階段中，這個 harness 都必須存在。

以下是 harness 在新 SDLC 的不同階段中，如何主動發揮作用：

1. 需求、規劃與架構（設定 Harness）

這是設定與調校 harness 的階段。在 AI 撰寫任何生產環境程式碼之前，開發者必須先設定好 Agent 的環境。

  • Harness 設定（Harness Configuration）：提供指示與規則檔案（例如建立 AGENTS.md 並定義架構約束），讓 harness 載入並提供給模型使用。
  • 實際行動（The Action）：開發者定義 Agent 可存取的工具（如特定的 API 或資料庫綱要），並設定 Agent 不能違反的基本規則。

2. 實作（執行 Harness）

在實際編寫程式碼期間，harness 作為邊界，確保 AI 模型專注、安全且高效。

  • 使用的 Harness 組件：沙箱（Sandboxes）、執行環境（Execution Environments）與工具（Tools）。
  • 實際行動（The Action）：當模型產生程式碼時，它會在 harness 隔離的沙箱中執行。如果模型需要讀取檔案或搜尋網路，它會使用 harness 提供的工具。

3. 測試與 QA（回饋迴圈）

在 Agentic 流程中，測試高度依賴 harness 來促進自主的自我修正。

  • 使用的 Harness 組件：協調邏輯（Orchestration Logic）與防護欄（Guardrails）。
  • 實際行動（The Action）：當 Agent 撰寫一個函式時，harness 會提供執行環境（例如沙箱化的終端機）來執行自動化測試。如果測試失敗，協調邏輯會擷取該環境的錯誤輸出，並將其回傳給模型，要求其重試。Harness 正是建立此自動化「思考 → 行動 → 觀察」迴圈的關鍵。

4. 程式碼審查、部署與維護（觀測 Harness）

即使程式碼編寫完畢，harness 仍能確保 Agent 在線上或準線上環境中安全運行。

  • 使用的 Harness 組件：Hooks 與可觀測性（Observability）。
  • 實際行動（The Action）：harness 會運行確定性的 hooks（例如，如果 Agent 試圖推送寫死的密碼，則阻擋該次 commit）。此外，可觀測性層會追蹤 token 成本、延遲和 Agent 偏離情況，讓人類工程師能精確稽核 Agent 做出特定部署決策的原因。

從「vibe coding」過渡到「agentic engineering」並非僅僅取決於你使用的工具——開發者可以使用完全相同的 Agent 進行 vibe coding 或應用 agentic engineering。相反地，它取決於你有多刻意地去設定和應用 harness。Vibe coding 依賴最少或隱式的起手架，純粹以快速實作為目標。Agentic engineering 則依賴清晰且廣泛的 harness 抽象層，引導 AI 從最初的規劃文件一路走到生產環境監控。

這種刻意設定所帶來的影響是高度可衡量的。公開基準測試使 harness 帶來的效果變得具體。在 Terminal Bench 2.0 上，一個團隊在完全不改變模型的情況下，僅透過調整 harness，就將一個編碼 Agent 從 30 名之外提升到前 5 名。LangChain 的另一項研究中，在固定模型的情況下，僅透過微調系統 Prompt、工具以及底層的中介軟體，就將同一基準測試上的編碼 Agent 分數提升了 13.7 分。

對於在整個 SDLC 中導入 AI 的團隊來說，日常中的這項觀察至關重要：當 Agent 出錯時，第一直覺通常是怪罪模型。但更常見的情況是，失敗可以追溯到遺漏的工具、模糊的規則、缺失的防護欄，或者是充斥著雜訊的 context window。老實說，大多數的 Agent 失敗都是設定上的失敗。

開發者角色的演變：協奏者與編排者

隨著 AI 承擔越來越多實作工作，開發者的角色正在以既令人興奮又令人迷惘的方式轉變。我們認為，將開發者在其中流暢切換的兩種模式想像為「協奏者（conductor）」與「編排者（orchestrator）」是很有幫助的。

協奏者（The Conductor）：親自即時引導

在協奏者模式下，開發者與 AI 結對程式設計師（AI pair-programmer）進行實時協作。他們待在 IDE 中，看著程式碼產生，以 Prompt 和修正來引導 AI，並對編寫的內容保持細粒度的控制。AI 是一件強大的樂器，但開發者在主動引導著每一個動作。

當處理複雜邏輯、除錯棘手問題，或是在不熟悉的 Repo 中工作且開發者需要理解做出的每一項變更時，這種模式很常見。像是 GitHub Copilot、Google 的 Gemini Code Assist、Cursor 和 Windsurf 等工具，主要是透過行內補全、對話介面和原地編輯（edit-in-place）功能來支援這種模式。

對於擁有傳統工程背景的開發者來說，協奏者模式非常自然。它保留了許多工程師所看重的理解力與控制感。但風險在於它也可能成為瓶頸——如果開發者親自引導每一次按鍵，AI 帶來的吞吐量提升將會受到限制。

編排者（The Orchestrator）：非同步、多 Agent 委派

在編排者模式下，開發者在更高的抽象層級上運作。他們定義目標、將其指派給 Agent，並審查結果——但他們不會一行行看著程式碼產生。Agent 可能在背景並行工作在程式碼庫的不同部分。開發者定期檢查、審查產出，並提供方向修正。

這種模式適用於定義明確的任務，例如錯誤修復、依循既有模式的特徵（feature）實作、程式碼庫遷移以及測試生成。Google 的 Jules、GitHub Copilot 的 agent 模式、Cursor 的背景 agent 以及 Claude Code 等工具，都是透過非同步任務執行來支援此模式，且通常在能完全存取 Repo、建置工具和測試套件的沙箱環境中運作。

編排者模式需要不同的技能組合。它不需要對語法和語言慣用語有深入的專業知識，而是需要以下方面的強大能力：

  • 規格說明（Specification）：足夠精確地定義任務，使 Agent 執行時不致產生歧義。
  • 拆解（Decomposition）：將大型任務拆解為適合 Agent 執行的適當大小單元。
  • 評估（Evaluation）：快速評估 Agent 的產出是否符合品質標準。
  • 系統設計（System design）：設計能讓 Agent 保持高生產力的約束條件、測試以及回饋迴圈。

80% 問題

AI 輔助開發中一個揮之不去的挑戰是我們所謂的「80% 問題」：AI agent 可以快速產生功能（feature）約 80% 的程式碼，但剩下的 20%——邊緣案例（edge cases）、錯誤處理、整合點以及微妙的正確性要求——則需要目前模型通常缺乏的深入 context 知識。

AI 錯誤的本質已從簡單的語法錯誤演變為更隱蔽的概念性失敗：對商業邏輯的錯誤假設、未就含糊不清的需求尋求澄清、遺漏邊緣案例，以及造成微妙長期維護負擔的架構決策。這些錯誤之所以更難察覺，正是因為程式碼「看起來是正確的」，甚至可能通過基本測試。

最能有效應對這一挑戰的開發者採取了一種特殊的姿態：他們將 AI 用於其擅長的事物（快速實作規格明確的任務），同時將自己的注意力留給 AI 難以處理的事物（模糊的需求、架構權衡以及正確性驗證）。他們不試圖透過全盤接受 AI 產出的所有內容來變快。他們透過將專業知識集中在最關鍵的地方來變快。

要有效應對這 80% 問題，需要將正確的工具應用於正確的工作階段。以「Conductor」身分運作的開發者，所需的工具組不同於以「Orchestrator」身分運作的開發者。為了瞭解如何將這些運作模式對應到你的日常工作流程，我們必須根據 AI agent 的自主權和整合度，對當前的 AI agent 版圖進行分類。

實務中的 Coding Agent

如今，建構 Agent 的開發者大部分工作都在終端機中進行，通常使用自然語言，且通常有另一個 coding agent 在幫忙打字。這是全新的體驗。一年前，相同的任務意味著框架、SDK 和雲端控制台。對於希望在日常中使用 coding agent 的開發者，以及希望自己建構 Agent 的開發者而言，這些取代舊有模式的新模式都值得被清楚命名。

Coding Agent 在開發者日常工作中的定位

Coding agent 在日常工作中出現在三個地方。大多數開發者會同時使用這三種。

在編輯器中：在開發者打字時提供下一行建議的行內自動補全。原地解釋或修改程式碼的對話面板。IDE 內部的整個程式碼庫感知。這是大多數人首次在程式設計中接觸 AI 的地方，也是工作保持流暢（flow）的地方。例子包括 GitHub Copilot、Cursor、Windsurf、JetBrains AI Assistant。

在終端機中：開發者從命令列啟動的 coding agent，以純文字交付目標，並讓它在整個程式碼庫中工作。擁有完整的檔案系統存取權限、多檔案編輯能力、執行工具和測試並根據結果進行迭代的能力。這也是如今進行深度 vibe coding 的地方。例子包括 Antigravity CLI、Claude Code、Codex CLI、Open Code 和 Cline。

在背景中：接收任務並在雲端託管沙箱中自主運行的 Agent，通常會執行數小時，且通常會產出 PR 作為產出。開發者將任務交派出去，稍後再進行審查。例子包括 Google Jules、GitHub Copilot 的 agent 模式、Cursor 的背景 agent，以及 Google 用於設計先進演算法的專用 AlphaEvolve agent。

在實務上，當開發者正在編寫程式碼，且希望在不中斷工作流的情況下獲得建議、快速編輯或解釋時，編輯器 Agent 會有所幫助。終端機 Agent 適合多檔案工作、探索陌生的程式碼庫，以及 Agent 需要執行程式碼並對其觀察結果做出反應的任務。背景 Agent 則適合開發者用一個段落即可描述清楚且可以暫時離開的明確任務，例如修復已知錯誤、生成測試套件，或將程式碼從一個框架遷移到另一個框架。同一個開發者在一天之內通常會同時使用這三種。

合適的起點取決於任務，而不是哪個類別在某個自主權階梯上排得最高。

Vibe Coding 生產環境等級的 Agent

目前為止討論的內容都是關於使用 coding agent 來開發軟體：撰寫功能、修復 bug、生成測試、重構程式碼。但是，當你需要開發的東西本身就是一個 Agent 時，會發生什麼事？

處理退款請求的客服機器人；交叉比對來源並產出有憑有據報告的調查助理；監控合規性並標記異常的內部工具。這些都不是在終端機（terminal）裡用一個 coding agent 就能解決的任務。它們是需要有自己的 tool、自己的 memory、自己的 evaluation 以及自己的部署基礎設施的產品。

以往用來產出原型腳本、基於終端機的相同工作流，現在也延伸到了這些 production agent。開發、評估和部署一個能大規模運作、具備 persistent memory、治理與可觀測性（observability）的真實 Agent，已從框架和雲端主控台（console）的任務，轉變為在同一個終端機中發生的事，且通常是透過與開發者原本就在使用的同一個 coding agent 對話來完成。

這個工作流在開發者需要一個能為真實使用者穩定運作的 Agent 時至關重要：跨 session 的 persistent memory、對 tool 和資料的 scope 限制權限、在出貨前捕捉 regression 的 eval 覆蓋率，以及追蹤 Agent 實際執行軌跡的可觀測性。對於一次性腳本或個人自動化，一般的 coding agent 就足夠了；此時 Agent 就是終點。但對於要大規模服務真實使用者的 Agent 來說，Agent 本身就是產品，它需要底層的基礎架構（substrate）支援。

Google 的 Agents CLI 就是圍繞這個概念構建的。它是一個小巧的命令列工具，打包了一套在 Google Cloud 上建構 Agent 的 skill，更重要的是，它可以搭配開發者偏好的任何 coding agent（如 Claude Code、Codex 或其他工具）一起使用。在一次性安裝之後，coding agent 就會獲得覆蓋完整 ADK 生命週期的七個新 skill：起手架專案、撰寫 agent 程式碼、進行評估、部署到 Agent Runtime 以及設定可觀測性。開發者不需要學習新的 SDK。他們只需描述自己的需求，coding agent 就會利用這些 skill 在每個步驟執行正確的操作。

具體來說，整個「建置－評估－部署」的循環看起來像這樣：

# One-time setup
uvx google-agents-cli setup

# Then in your coding agent:
> Build a support agent that answers questions from our docs.
> evaluate it on the FAQ dataset
> Deploy it to Agent Engine

程式碼片段 1：Agents CLI 的設定與建置。

在這一句簡單的指令背後，coding agent 會從範本起手架一個專案、撰寫 ADK 程式碼、生成 evalset、對 Agent 進行測試、部署到 Agent Runtime 並回報結果。對於偏好直接操作的開發者，相同的操作也可以透過簡單的 CLI 指令來完成（例如 agents-cli create、agents-cli playground、agents-cli eval、agents-cli deploy）。

過去，production agent 需要與原型不同的獨立技術堆疊和工作流。現在，昨天還在開發者筆電上運行的原型，今天就能直接變成服務真實使用者的 production agent，無需重寫。

同樣的工作流可以從單一 Agent 擴展到多個 Agent。ADK 提供了基於圖（graph）的工作流、用於建置協同運作 Agent 的 multi-agent 工作流，以及共享 session 狀態、LLM 驅動的委派（delegation）和顯式呼叫（explicit invocation）等互動機制，這些都可以組合成適合該問題的任何 multi-agent 模式。

跨 Agent 的協調在簡單情況下可以透過共享 session 狀態完成，在需要存取 tool 時則透過 Model Context Protocol（MCP）進行，而跨 Agent 的委派則透過 Agent2Agent（A2A）協定來處理。Anthropic 的工程團隊在 2026 年初發表了一項實驗，在該實驗中，運行於此類架構上的 Agent 團隊在兩週內用 Rust 寫出了一個 C 編譯器，人類只負責設定方向和審查輸出，而不動手撰寫實作程式碼。瓶頸從「撰寫程式碼」轉移到了「定義程式碼應該做什麼」並「驗證 Agent 是否確實完成」。

對於開發者來說，這在實務上的意義非常簡單：今天用來產出腳本的同一個 vibe coding 工作流，明天就能產出 production agent。建置、評估、部署、觀測、微調的整個生命週期都整合在同一個地方。從點子到運行中 Agent 的路徑已從數週縮短到數小時，且大部分的工作現在都以自然語言進行。

讓這個工作流在團隊規模下達到 production-grade 的實踐方法——從 spec 驅動開發（spec-driven development）、結構化程式碼審查（code review）到安全防護欄（guardrails）、沙箱化（sandboxing）和零信任開發，都收錄在第 5 天的伴隨論文《Vibe Coding 時代下的 Spec 驅動 Production-Grade 開發》中。

AI 開發的經濟學

當評估 AI 對軟體開發生命週期（SDLC）的影響時，討論往往始於也終於開發者速度（velocity）：我們能多快寫出程式碼？然而，對於工程主管來說，更關鍵的指標是整體擁有成本（Total Cost of Ownership, TCO）。

為了理解 AI 輔助開發的真實成本，我們必須檢視不同的工作流如何將財務與營運負擔在資本支出（CapEx，即建置某樣東西的預期前期投資）與營運費用（OpEx，即運作、修復和維護它的持續成本）之間轉移。至關重要的是，在 AI 時代，OpEx 在很大程度上受到 token 經濟（token economy）的支配。

Vibe Coding 的隱藏債務（低 CapEx、高 OpEx）

乍看之下，vibe coding 似乎非常劃算。進入門檻幾乎為零：只需要訂閱 AI 助理的標準月費，並輸入幾次隨意的 prompt。CapEx 微乎其微，因為開發者完全依賴模型的基準能力，而不是花時間在系統設計上。

然而，vibe coding 的經濟模型背後隱藏著龐大且持續累積的 OpEx 負擔：

  • Token 消耗率（Token Burn Rate）：與 LLM 的每次互動都會根據輸入和輸出 token 產生費用。在 vibe coding 中，開發者經常將龐大、未整理的檔案直接丟進 context 視窗中，並反覆要求模型修復自己那些未經驗證的錯誤。這會產生一個昂貴的「prompting 迴圈」，消耗大量 API token，且首次成功率（first-pass success rate）極低。
  • 維護稅（Maintenance Tax）：透過臨時性（ad-hoc）prompting 撰寫的程式碼通常缺乏結構一致性。當六個月後出現 bug 時，人類工程師必須花費數天去對那些沒有結構、AI 生成的「義大利麵條式（spaghetti）」程式碼進行逆向工程。
  • 安全補救（Security Remediation）：在沒有自動化評估 harness 的情況下，快速生成程式碼會導致漏洞的快速累積。在 production 中修復安全漏洞的成本，呈指數級高於在設計階段就抓到它。

Agentic Engineering 的投資（高 CapEx、低 OpEx）

Agentic Engineering 翻轉了這種經濟模型。它要求在生成任何一行 production 程式碼之前，先投入深思熟慮的前期工程時間和資源。

Agentic Engineering 的 CapEx 包含設計 API schema、建置確定性的測試套件（test suite），以及最關鍵的：結構化 Agent 的 context。雖然前期的成本較高，但出貨和維護功能的邊際成本會大幅降低。AI 會在一個受到嚴格治理的「工廠」中運作，這意味著它的輸出在結構上是健全的、經過預先測試的，並且符合公司的標準。

將 Context Engineering 視為財務槓桿

在 token 經濟中，context engineering 不僅僅是一項技術技能，更是一項財務策略。LLM 是針對你傳送給他們的每條資訊進行收費。在規模化運作時，每次 prompting 都將整個 100,000 token 的 repo 丟進去，在財務上是行不通的。

有效的 context engineering 可以確保模型接收到高密度、高訊號的資料（例如精確的 AGENTS.md 檔案和架構防護欄），而不是雜亂、龐大且充斥雜訊的內容。藉由在前期提供正確的 context，開發者可以大幅提升 Agent 的首次成功率，避免困擾 vibe coding 的高成本嘗試與出錯迴圈。

透過動態 Context 與 Skill 來擴展效率

為了真正最佳化 OpEx，進階的 Agentic Engineering 依賴於動態 context，這是透過使用「skill」或 tool 呼叫（例如 Model Context Protocol 伺服器）來實現的，我們在第 3 天的論文中會詳細介紹這部分。

智慧模型路由

此外，Agentic Engineering 允許進行智慧模型路由（intelligent model routing）。在 vibe coding 工作流中，開發者通常會為每次互動依賴單一且龐大的尖端模型（frontier model）——支付昂貴的 token 費用，僅為了讓 AI 修正一個拼字錯誤或生成一個基本的單元測試。

一個精心設計的工廠模型（factory model）能避免這種浪費。它在高度複雜的任務（如 Requirements、Architecture 和初始 Implementation）中使用大型且先進的模型，但會自動將確定性、複雜度較低的任務（如 Test Generation、Code Review 和 CI/CD 監控）路由到較小、較快且顯著便宜的模型。藉由協調一個多模型生態系（multi-model ecosystem），工程團隊可以維持巔峰的輸出品質，同時有系統地降低營運的 token 成本。

從哪裡開始

從語法（syntax）到意圖（intent）的轉變並非未來的狀態，而是我們今天眼前的工作。不論你是一位獨立開發者，還是正在思考團隊或組織該如何導入這些工具的主管，底層的原則都是相同的：AI 會放大它所引入的工程文化。以下實踐方法將此原則轉化為實際行動。

給個人開發者

  1. 為專案設定一個 AGENTS.md（或等效檔案）。選擇與你所選 coding agent 相符的規範。從 10 行開始：技術堆疊（stack）、規範（conventions）、硬性規則（hard rules）、工作流（workflow）。每當 Agent 做了不該再犯的事情時，就加上一條規則。
  2. 為你的 coding agent 安裝一組 skill（例如 Agents CLI），用來建置、評估、部署和最佳化 Agent。
  3. 挑選一個重複性的工作流，並將其製作成你的第一個 Agent。例如：研究工作流、code review 流程、定期報告、固定產出的內容。使用 coding agent 做出原型，當它證明了自己的價值後，再透過 Agents CLI 將它升級為 production agent。完整地從頭到尾建立一個 Agent，學到的東西比讀一百篇相關文章還多。
  4. 在生成程式碼之前先撰寫測試與 eval。兩者結合就是與 AI 的契約。寫得好的測試與 eval 套件比任何自然語言 prompt 都能更精確地傳達意圖，並將 AI 輔助開發從 vibe coding 轉化為 Agentic Engineering。
  5. 審查 Agent 所生成且準備要出貨（ship）的每一行程式碼。對於任何看起來很聰明的寫法保持懷疑。檢查 import 的是否為真實套件。驗證錯誤處理（error handling）是否涵蓋了實際的失效模式（failure modes）。團隊看不懂的程式碼，最後會變成團隊無法承受的除錯成本。
  6. 維持你的開發者技能。AI 負責處理例行公事，好讓開發者專注於有挑戰性的任務。這種安排只有在基礎技能——如除錯（debugging）、系統設計（system design）、對效能與正確性的直覺——保持敏銳的情況下才有效。把 AI 當作是在更大規模上發揮專業能力的方式，而不是專業的替代品。持續進行複雜的除錯練習、對 AI 的輸出進行 code review，以及參與架構討論，對於成長為一名工程師仍然至關重要。

給工程主管

  1. 將 context engineering 視為團隊中的一等公民工程實踐（first-class engineering practice）。將 AGENTS.md、system prompt、eval 套件和 skill 函式庫視為程式碼來對待：在 PR（pull request）中進行審查、隨專案進行版本控制、並由特定的工程師負責維護。如果沒有這種紀律，harness 就會失焦，且 Agent 的行為在團隊中將變得無法重現。
  2. 以 eval 為標準，而非以 demo 為標準。一個可以運行的 demo 只能證明 Agent 成功過一次；而通過 eval 套件則能證明它運作穩定可靠。但是，沒有清晰量規（rubric）的 eval 是測不出任何東西的。定義你評分的項目：任務成功率、tool 使用品質、軌跡合規性（trajectory compliance）、幻覺（hallucination）以及回應品質。要求將具備明確量規的 eval 覆蓋率作為任何 Agent 進入共享工作流的先決條件，就像測試覆蓋率用來管控服務部署一樣。
  3. 重新塑造針對 AI 生成程式碼的 code review。AI 生成的程式碼需要與人類撰寫的程式碼相同、甚至更高程度的審查，並要特別注意憑空捏造的依賴項（hallucinated dependencies）、不足的錯誤處理，以及乍看之下正確但實際上微妙的正確性缺陷。培訓審查人員了解生成程式碼的失效模式，並相應地調整審查清單。
  4. 在團隊規範中區分原型開發與 production 開發。Vibe coding 是適合探索的速度；Agentic Engineering 則是適合 production 的紀律。讓邊界清晰明確：哪些專案、哪些 branch、哪些環境適用哪種工作模式。邊界模糊的團隊，往往會不小心將原型直接出貨上線。
  5. 投資 harness 組件作為團隊的共享資產。可重複使用的 system prompt、skill 函式庫、MCP 伺服器連接以及評估 harness，會在各專案之間產生加乘效果。將它們視為基礎設施：有文件記載、有專人維護，並且刻意地去改善。能從 AI 輔助開發中獲得最大價值的團隊，通常是那些「建置一次 harness，並進行多次微調」的團隊。

給組織

  1. 將 AI 輔助開發視為一種工程投資，而非僅是生產力功能。獲得最大收益的團隊，會將 AI 工具與 eval 覆蓋率、可觀測性以及明確的架構標準相結合。如果沒有這些起手架就直接推出 coding agent，只會帶來沒有品質的速度，這會以任何團隊都無法償還的速度累積技術債。
  2. 在規模化之前先投資 production 基礎架構（substrate）。在筆電上用 vibe coding 寫出來的原型並不是 production 系統。將原型升級為 production 系統的關鍵，在於圍繞著它的一套維運紀律：在 CI 中運行軌跡（trajectory）與最終回應的 eval、每一次 Agent 運作的 trace 紀錄、每個 Agent 的 scoped 權限限制，以及針對生成程式碼的失效模式進行安全審查。在第一個 production agent 出貨之前就建置好這個基礎架構，而不是事後才做。
  3. 採用工具與 Agent 間通訊的開放標準。用於 tool 存取的 Model Context Protocol（MCP）以及用於跨 Agent 委派的 Agent2Agent（A2A）協定，正逐漸匯聚成為 multi-agent 系統的聯絡組織。現在選擇它們，能保留混用不同廠商與框架的彈性，避免日後需要重新進行平台轉移（re-platforming）。
  4. 規劃人類與 Agent 協同的混合團隊，而非僅限人類或僅限 Agent 的工作流。過去一年中最強大的 production 成果，來自於「人類設定方向、Agent 負責實作，並由清晰的交接協定（handoff protocols）管控邊界」的架構。Code review 流程、on-call 輪班制度以及團隊結構都需要隨之演進，以反映 Agent 現在是參與者，而不僅僅是工具。
  5. 重新圍繞著判斷力（judgment）而非僅僅是實作（implementation）來重新定義招募與技能培訓。隨著實作變得更快、更自動化，瓶頸轉移到了規格制定（specification）、評估（evaluation）、架構判斷與審查。刻意去招募和培訓這些技能。未來幾年內最有價值的工程師，將是那些能夠良好指導 Agent 的人，而不是那些能寫最多程式碼的人。

結論：意圖是新的介面

從語法（syntax）到意圖（intent）的轉變不是對未來的預測，而是眼前的現實。開發者花在描述他們「想要什麼」的時間，已經多過於指定「如何建置它」。SDLC 已經圍繞著 AI 能力進行了壓縮、重構與重新想像。問題不在於這個轉型是否會發生，而是個人開發者、團隊和組織能多有效地駕馭它。

我們在這篇論文中提出的框架——從 vibe coding 到 Agentic Engineering 的光譜、開發者角色的「指揮者（conductor）到編排者（orchestrator）」模型、ambient、workflow 以及自主性（autonomous）Agent 的分類法，以及軟體生產的工廠模型（factory model）——為理解這個快速演進的領域提供了一套心智模型。即使具體的工具與能力不斷演進，這些模型依然適用。

有三個原則歷久彌新：

  1. 結構能規模化，感覺（vibes）不行。Vibe coding 是探索、原型開發和個人專案的有效方法。但對於組織所依賴的軟體，Agentic Engineering 的紀律——規格說明、測試、防護欄以及人類對架構的監督——並非選配。「看起來好像能跑」與「在所有條件下都能正確運作」之間的落差，正是 production 服務中斷、安全漏洞和維護噩夢的溫床。
  2. AI 會放大你的工程文化。擁有強大測試實踐、清晰架構標準以及健康 code review 流程的組織，從 AI 輔助開發中獲得的價值會顯著多於沒有這些實踐的組織。AI 是一個乘數——它會同時放大你的優勢和劣勢。
  3. 人類的角色正在演進，而非消失。理解架構、能定義精確規格、批判性評估輸出，並能設計出有效約束與回饋迴圈系統的開發者，比以往任何時候都更加珍貴。重要的技能正在從實作轉移到判斷力，從撰寫程式碼轉移到設計「生產程式碼的系統」。

我們正處於一場變革的開端，這場變革不僅將重塑軟體的建置方式，還會重塑我們能建置出什麼樣的軟體。更小的團隊將能夠解決更大的問題。個人開發者將能夠建置並維護以往需要整個部門才能搞定的系統。開發軟體的門檻將持續降低，將軟體開發的實踐開放給更廣泛的人群。

蓬勃發展的團隊將會是那些在擁抱 AI 作為強大工具的同時，依然維持身為可靠軟體基石的工程紀律的團隊。他們會明白軟體工程的未來不是在人類專業與 AI 能力之間做選擇，而是設計出兩者都能貢獻其獨特優勢的系統。

生成（Generation）問題已被解決。驗證（Verification）、判斷（judgment）與方向引導（direction）是新的工藝。
