# 當你的鍵盤不再是瓶頸：Vibe Coding 到 Agentic Engineering 的典範轉移

> 來源：Kaggle × Google「5-Day AI Agents Intensive Course」Day 1 白皮書同步 Podcast  
> 影片連結：https://www.youtube.com/watch?v=cbzmr7vt4XA  
> 主題：Introduction to Agents and Vibe Coding

---

## 你以為在「用 AI 寫程式」，其實你在為技術債的複利繳利息

41%。

這是 2026 年初所有新程式碼中，由 AI 全自動生成的比例。加上有 85% 的專業開發者每天都在使用 AI 輔助工具——這個數字不只是「AI 很厲害」這麼簡單，它意味著軟體開發的底層假設，已經悄悄被改寫了。

問題是，大多數人感覺到改變，但沒有看清楚改變的方向。他們以為，只要把 AI 接進工作流程，就算轉型了。但真正的分水嶺，不是「你有沒有在用 AI」，而是「你用 AI 的方式，是在加速建造，還是在加速累積債務」。

這個差別，有一個名字：**Vibe Coding vs. Agentic Engineering**。

---

## 差別不在工具，在於你如何驗證結果

Vibe Coding 這個詞由 Andrej Karpathy 在 2025 年初提出。它描述的是一種完全順著「感覺走」的工作方式：你用自然語言描述你想要什麼，接受 AI 吐出來的任何程式碼，如果壞了，就把錯誤訊息直接貼回去，叫它再試一次。整個過程就像是即興演奏——速度極快，摩擦極低，但沒有任何保障。

Vibe Coding 最大的誘惑，是它的零門檻啟動成本。打開聊天視窗，輸入一句話，馬上有東西跑出來。感覺像魔法。

但 Agentic Engineering 告訴你：魔法是有代價的。

在 Agentic Engineering 的框架裡，AI 一樣負責實作，但它必須在嚴格的約束條件下運作：**正式規格（formal specifications）**、**自動化測試套件（automated test suites）**、**持續整合閘門（CI gates）**。這些「閘門」是自動化的品質檢查點，任何程式碼在進入正式環境之前，都必須通過它們。

最關鍵的一句話：**兩者都在用 AI，差別完全在於如何驗證輸出結果。**

Vibe Coding 的驗證方式是眼測：「按下去，畫面有動，應該可以吧。」Agentic Engineering 的驗證方式是系統性的：用硬測試（deterministic tests）確認程式碼行為，用系統性評估（evals）驗證 AI 推論過程的品質。

沒有系統性驗證，你永遠都只是在 vibe coding——不管你的初始 prompt 有多精心設計。

### Michelin 星餐廳的啟示

Podcast 裡有個比喻讓這個差異變得非常具體：

Vibe Coding 像是週末自家烤肉。你隨性作，漢堡烤焦了就刮掉繼續吃，誰也不會告你。驗證標準是「好像可以吃」。

Agentic Engineering 像是米其林星級廚房。每種食材精確到公克，所有肉品都要測量中心溫度才能出廚房。因為你面對的是付費客人，一次失誤就是商譽災難。

如果你告訴你的 CTO，你們的金流系統正在 vibe coding，他當場就會叫你走人。但如果你說，你們採用 Agentic Engineering，用自動化 evals 和安全閘門驗證每一次 AI 輸出，他聽到的是一個可擴展的現代工作流程。

## 真正的核心技能，不是 Prompt，是 Context

許多人以為，AI 時代的關鍵技能是「會下好 prompt」。這個認知已經過時了。

現在更精確的說法是：**Context Engineering（情境工程）**。

Prompt 工程是試圖透過措辭技巧讓模型給出更好的答案，這是在跟一個固定的工具搏鬥。Context Engineering 則是把正確的資訊，以正確的結構，在正確的時機交給 AI——讓它一開始就有能力做出好的判斷。

要讓 AI 不幻覺、不偏軌，你需要精確平衡六種 Context：

| 類型 | 作用 |
|------|------|
| **Instructions（指令）** | 定義 Agent 的核心邊界與行為人格 |
| **Knowledge（知識）** | 架構圖、API 文件、Style Guide——Agent 需要了解的工作環境 |
| **Memory（記憶）** | 短期工作日誌 + 長期專案狀態，防止 Agent 每次都從零開始重試 |
| **Examples（範例）** | 從程式庫中取出的參考程式碼模式，讓 AI 模仿，而非憑空發明 |
| **Tools（工具）** | Agent 實際被允許操作的 API、CLI、檔案系統 |
| **Guardrails（護欄）** | 攔截器——當 AI 要做出危險操作時（例如刪除正式資料庫），直接在執行前阻擋 |

### Context Rot：更多資訊，反而更差的輸出

一個常見的直覺錯誤是：既然 Context 這麼重要，那就把所有東西都放進去。把整個 Repo、所有架構圖、所有規則，一股腦塞進 prompt。

這個做法有個致命的後果：**Context Rot（情境腐化）**。

現代 LLM 確實有數百萬 token 的超大 Context Window，但它的注意力機制（attention mechanism）在被噪音淹沒時會稀釋。你把一百萬 token 的不相關程式碼塞進去，AI 反而失去焦點——它可能會忘記主要指令，或者從完全不相關的資料夾裡抓一個變數，因為它被龐大的資訊量搞混了。

**更多資訊，實際上降低了輸出的智能水準。**

解法是「動態 Context」——按需載入，用完即棄。最強的模式是 **Agent Skills**：與其在一開始就告訴 AI 所有事，不如給它一份輕量的技能選單。當任務需要資料庫存取時，Agent 才觸發資料庫技能，載入相關的 schema 和工具，完成工作後再放下那份 Context。

背包的比喻很精準：靜態 Context 是你背包裡的帳篷和水，是你永遠必須帶著走的東西。動態 Context 是某段路程需要的那張精細地形圖，你只在到達那個路口的時候才把它拿出來，然後收回去。這樣你才不會走一整段山路都在盯著一張只有 200 公尺用到的地圖。

## SDLC 的崩塌與重建

當開發者的時間不再花在打字，而是花在設計 Context——整個軟體開發生命週期（SDLC）的節奏就徹底變了。

**實作**從幾週壓縮到幾小時。這不是誇大，是因為打字速度本來是主要瓶頸，而現在這個瓶頸消失了。

但人類的瓶頸沒有消失，只是**移到了週期的兩端**：

- **前端**：需求收集不再是一份靜態 PDF 規格書，而是跟 AI 即時對話，10 分鐘生成一個運行中的原型，然後用這個具體的原型去精煉最終的規格需求。
- **後端**：測試不只是檢查最終輸出。你現在需要評估「軌跡」——AI 中途有沒有用了批准的工具？修改敏感檔案之前有沒有先確認權限？

一個流暢運行、語法完全正確的程式碼，如果中途偷偷繞過了關鍵的安全檢查——它實際上比一個明顯噴出 syntax error 的程式碼更危險。

**你必須給 AI 的數學過程打分，不只是給最終答案打分。**

甚至連維護都被升級了：那些每個人都怕碰、原始開發者早就離職的「放射性義大利麵條程式碼（radioactive spaghetti code）」——現在可以被 AI Agent 系統性地梳理和重構了。

## Agent = Model + Harness，而 Model 只佔 10%

這是整個 Podcast 最反直覺的數字：**一個 AI Agent 的能力，只有約 10% 來自底層的語言模型本身。**

剩下的 90%，是 Harness。

Harness 是「圍繞著模型的鷹架系統」，它讓模型的原始智能不會迷失在懸崖邊。它包含：

- **沙箱（Sandboxes）**：隔離環境，讓 AI 可以真正執行和測試剛寫好的程式碼，同時不會炸掉你的正式伺服器。
- **協調邏輯（Orchestration Logic）**：把一個大任務分解、路由給各個專門的子 Agent。
- **可觀測性基礎設施（Observability）**：日誌、執行軌跡、成本計量——讓你隨時知道 Agent 是在成功推進，還是陷入「Agent Drift」（它開始鑽入一條錯誤邏輯的兔子洞，試圖寫幾千行無用的程式碼來解決一個它其實完全不理解的錯誤）。

Podcast 引用了 Terminal Bench 2.0 的基準測試數據：有一個工程團隊，**在完全不動底層模型的情況下，只靠優化 Harness，就把他們的 Coding Agent 從排行榜前 30 名外直接推進到前 5 名**。

這個數字告訴我們：當你的 AI 工具一直幻覺、一直輸出壞掉的程式碼，你的第一反應不該是「這個模型太爛了，換一個吧」。你應該照照鏡子，審視你自己建的 Harness：你給它的規則夠清楚嗎？你的工具定義夠精準嗎？你忘了哪個護欄？

模型是引擎，Harness 提供的是變速箱、方向盤和煞車。

## 指揮者與編排者：兩種截然不同的工作模式

掌握了強大的 Harness 之後，開發者的日常工作分裂成兩種模式：

**指揮者模式（Conductor Mode）**  
手動、即時、親密協作。你坐在 Cursor 或 GitHub Copilot 前，逐行引導 AI。幾乎是即時的次秒反饋。在複雜邏輯的除錯、或深度定制的程式碼庫導覽時，指揮者模式最能發揮人類直覺的價值。你握著指揮棒，讓每一個音符即時演奏。

**編排者模式（Orchestrator Mode）**  
非同步、高層次委派。你不看著程式碼一行一行出現，你直接指派一個大目標給在背景運行的終端 Agent——例如 Google Jules。「把這個資料庫從 v3 遷移到 v4」——然後 Agent 在沙箱裡獨立工作，三小時後你去看 Pull Request。對大規模測試生成、跨多檔案重構來說，編排者模式的威力難以估量。

在這兩種模式之間流暢切換，是開發者處理「80% 問題」的關鍵。

**80% 問題**指的是：AI 可以迅速生成任何功能大約 80% 的實作。樣板程式碼、標準架構模式，AI 完全統治。但最後那 20%——奇怪的邊界情況、高度特定的業務邏輯、微妙的系統整合——需要的是人類深度的情境知識，是目前的模型還不具備的能力。

這個框架，也徹底瓦解了「AI 搶走開發者工作」的焦慮論述。AI 接管的是機械性的 80%。人類的認知能量，則全部釋放出來，集中在最具價值的 20%——那些真正需要架構判斷力的地方。

## Token 經濟學：Vibe Coding 是一張高利率信用卡

這套方法論背後還有一個非常務實的財務邏輯。

Vibe Coding 的成本結構：**極低的啟動成本（CapEx），但高得可怕的運行成本（OpEx）**。打開聊天視窗，零前期投資。但你在用試誤法燒 API Token，把非結構化的龐大檔案塞進 Context Window，AI 幻覺了你再 prompt，再幻覺，再 prompt——這是無止境的 Token 消耗。加上沒有自動化測試，你正在建造一座義大利麵條碼塔，它最終會在自己的重量下崩塌，需要昂貴的人工介入去解糾。

Vibe Coding 是把整個開發週期放在高利率信用卡上：立即的原型快感，但利息（安全漏洞、技術債、Token 燒耗）會在後面讓你的時間表破產。

Agentic Engineering 是**高 CapEx、低 OpEx**：前期你必須認真投資——建 API schema、撰寫完整的 agents.md 規格文件、搭建沙箱、寫決定性的 evals。這是一筆重的頭期款。但一旦這座工廠建好，每新增一個功能的邊際成本就大幅下降。AI 第一次就能生出正確的程式碼，Token 消耗驟減，維護稅消失。

這是用付清利息的固定利率房貸取代循環信用卡。前期更難，但財務上永遠可持續。

還有一個優化工具：**智慧模型路由（Intelligent Model Routing）**。Vibe Coding 的開發者傾向把所有問題都丟給最貴的旗艦模型，包括「幫我格式化這個字串」這種小事。成熟的工廠模型裡，Harness 會扮演聰明的交通指揮員——複雜的架構推理走旗艦模型，簡單的單元測試撰寫或 JSON 格式化走輕量快速的小模型。這個單純的架構決策，可以讓運行成本降低幾個數量級，同時維持完全相同的程式碼品質。

## 尾聲：當 AI 打好了所有磚塊，誰來教下一代怎麼蓋房子？

Podcast 在最後拋出了一個沒有標準答案的問題，留給聽眾回去想：

如果 AI 現在負責鋪所有的磚——承擔了所有機械性的 80% 實作——而人類的價值完全集中在高層次的架構判斷上；那麼，下一代的初級開發者，如果從來不需要自己動手鋪磚，他們要從哪裡學到那個架構判斷力？

這個「師徒斷層（mentorship gap）」，可能是未來十年軟體工程最核心的挑戰。

從 Autocomplete 到 Agentic Engineering，我們確實走過了一條讓程式設計越來越接近人類思維的路。但每一次讓機器做更多，我們都在隱性地問自己：這個被自動化掉的過程，曾經是在教我們什麼？
